# BLO-063: RAG — similar code search

## Objective

Implement the `rag_search_similar` MCP tool that searches the vector database for SM83 assembly patterns similar to a given block. Done when: a user can submit an assembly block and receive the top K most similar annotated matches with similarity scores, annotated source, and original project references.

## Context

Part of the Phase 7 RAG system described in `docs/mcp-spec.md`. Once the reference database is populated (BLO-065), this tool enables the core use case: given an unknown assembly block from a ROM being decompiled, find similar patterns from known decompilation projects (pokered, pokecrystal, etc.) to assist with annotation and understanding. This is the primary search interface for the RAG system.

## Acceptance Criteria

- [ ] `rag_search_similar(assembly_block, top_k?)` MCP tool registered and functional
- [ ] Input assembly block is canonicalized using the same pipeline as indexing (from BLO-065)
- [ ] Returns top K matches (default K=5) ordered by similarity score
- [ ] Each result includes: similarity score, annotated assembly source, raw assembly, source project name, and metadata
- [ ] Handles edge cases: empty DB, no matches above threshold, malformed input
- [ ] Tool returns structured error responses following project convention (`{"error": "CODE", "message": "..."}`)
- [ ] Domain logic in `domain/` with no project imports; tool handler is thin wrapper in `tools/`
- [ ] Tests written (TDD) for search, ranking, result formatting, and edge cases

## Technical Approach

1. **Canonicalization reuse**: Use the same instruction canonicalization from BLO-065's domain module to normalize the query assembly block before embedding.
2. **Vector search**: Query ChromaDB with the embedded canonicalized block. Use cosine similarity (ChromaDB default). Return top K results.
3. **Result formatting**: Each match should include the similarity score (0.0-1.0), the annotated assembly from the reference pair, the original raw assembly, the source project (e.g., "pokered"), and any stored metadata (file path, function name).
4. **Threshold filtering**: Optionally filter results below a minimum similarity threshold to avoid returning noise.
5. **Key files**:
   - `src/blobert_mcp/domain/rag/search.py` — search logic, result ranking
   - `src/blobert_mcp/tools/rag_search_similar.py` — MCP tool handler
   - `tests/domain/rag/test_search.py` — unit tests
   - `tests/tools/test_rag_search_similar.py` — tool handler tests

## Notes

- Search quality depends heavily on the canonicalization strategy from BLO-065. If results are poor, the canonicalization may need tuning.
- Consider adding a minimum similarity threshold parameter in addition to top_k.
- Large assembly blocks may need to be chunked or windowed for effective similarity search — evaluate during implementation.
- The embedding model choice (from BLO-065) directly affects search quality here.
