# BLO-047: Persistent save states

## Objective

Replace the current in-memory BytesIO save state storage with disk-backed persistent storage, so that save states survive server restarts. Done when save states are written to disk alongside the ROM file, include JSON metadata sidecars, the existing MCP tool API (`gb_save_state`, `gb_load_state`, `gb_list_states`) remains unchanged, and the implementation has full test coverage.

## Context

Save states currently use in-memory `BytesIO` objects, which are lost when the MCP server restarts. For longer decompilation sessions that span multiple server invocations, losing save states forces the LLM to re-navigate to previously explored game states. Disk-backed storage solves this by persisting states in a directory structure alongside the ROM file, similar to how the knowledge base SQLite file is stored. This is a storage backend enhancement -- no new MCP tools are added. The existing `gb_save_state`, `gb_load_state`, `gb_list_states`, and `gb_delete_state` tools continue to work with the same interface.

## Acceptance Criteria

- [ ] Save states are written to disk as `.state` files in a dedicated directory alongside the ROM file
- [ ] Each save state has a JSON metadata sidecar (`.meta.json`) containing: state_id, name, frame_count, PC, timestamp, ROM hash
- [ ] Save state directory structure: `<rom_name>.blobert/states/<state_id>.state` + `<state_id>.meta.json`
- [ ] `gb_save_state` writes to disk instead of BytesIO; returns the same state_id format
- [ ] `gb_load_state` reads from disk instead of BytesIO; same behavior on success/failure
- [ ] `gb_list_states` reads metadata from sidecar files; includes states from previous sessions
- [ ] `gb_delete_state` removes both `.state` and `.meta.json` files from disk
- [ ] States from a different ROM (mismatched ROM hash) are not loaded; returns clear error
- [ ] Migration path: if any in-memory states exist at save time, they are flushed to disk
- [ ] Existing MCP tool API signatures are unchanged -- this is a transparent backend swap
- [ ] Domain logic is TDD -- tests written before implementation
- [ ] All existing save state tests continue to pass (or are updated to reflect disk storage)

## Technical Approach

1. **Storage directory** -- Create `<rom_name>.blobert/states/` directory next to the ROM file when the first save state is created (follows the pattern of the KB database stored alongside the ROM).
2. **Save** -- `gb_save_state`:
   - Generate state_id (existing logic)
   - Write PyBoy state to `<state_id>.state` file via `pyboy.save_state(file)`
   - Write metadata to `<state_id>.meta.json`: `{state_id, name, frame_count, pc, timestamp, rom_sha256}`
3. **Load** -- `gb_load_state`:
   - Read `.meta.json` to verify ROM hash matches current ROM
   - Open `.state` file and pass to `pyboy.load_state(file)`
4. **List** -- `gb_list_states`:
   - Glob `*.meta.json` in the states directory
   - Parse and return metadata for all states (sorted by timestamp)
5. **Delete** -- `gb_delete_state`:
   - Remove both `.state` and `.meta.json` files
6. **Domain layer** -- Add a `SaveStateStore` class (or similar) in domain that handles file I/O, metadata serialization, and ROM hash validation. EmulatorSession delegates to this instead of maintaining a `dict[str, BytesIO]`.
7. **ROM hash** -- Compute SHA-256 of the ROM file at load time; store in session for comparison during state load.
8. **Tests** -- TDD with `tmp_path` fixtures for disk operations. Test metadata round-trip, ROM hash mismatch rejection, listing across "sessions" (separate store instances pointing at the same directory).

## Notes

- The `.blobert/` directory pattern mirrors how the KB database is stored. If the KB uses a different convention, align with that.
- Consider filesystem atomicity: write to a temp file and rename to avoid corrupt states on crash.
- State files can be large (PyBoy states are typically 100-300KB). This is manageable for dozens of states but worth noting.
- The ROM hash prevents loading states from a different ROM version, which would cause undefined emulator behavior.
- Future enhancement: state compression (gzip) to reduce disk usage. Not in scope for this ticket.
