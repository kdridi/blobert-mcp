# BLO-004: Dynamic execution, input, save states & screenshot

## Objective

Implement the P0 tools for controlling the emulator at runtime: stepping, running to an address, reading registers, pressing buttons, managing save states, and capturing screenshots. After this ticket, the LLM can play the game, explore execution paths, checkpoint/restore state, and see the screen — enabling the full interactive exploration loop.

## Context

This is the dynamic counterpart to the static reading tools (BLO-002). These tools mutate emulator state — they advance execution, press buttons, and save/load states. Together with the disassembler (BLO-003), they complete the core exploration workflow from the spec.

Covers spec sections: §2 (Dynamic execution — P0 tools), §4 (Save states — P0 tools), §5 (Inputs — P0 tool), §6 (Visual — P0 tool).

## Acceptance Criteria

- [ ] `gb_step` — Advances execution by `count` frames (mode="frame") or instructions (mode="instruction"). Returns current PC, instruction, and full register state. Frame mode uses `pyboy.tick()`. Instruction mode uses hook-based stepping.
- [ ] `gb_run_until` — Executes until `target_address` is hit or `timeout_frames` exceeded. Uses `pyboy.hook_register()`. Returns registers at breakpoint or timeout error.
- [ ] `gb_get_registers` — Returns all CPU registers: A, B, C, D, E, F, H, L, AF, BC, DE, HL, SP, PC, and individual flags (Z, N, H, C).
- [ ] `gb_press_button` — Presses or releases a button (a/b/start/select/up/down/left/right). Validates button name.
- [ ] `gb_save_state` — Saves emulator state to in-memory BytesIO. Returns a state_id. Supports optional `name`.
- [ ] `gb_load_state` — Restores a previously saved state by state_id. Returns registers after restoration.
- [ ] `gb_screenshot` — Captures the screen as a base64-encoded PNG or WebP image. Supports `scale` parameter for upscaling.
- [ ] Save states are stored in `EmulatorSession.save_states` dict with auto-incrementing IDs.
- [ ] All tools return proper errors when no ROM is loaded.
- [ ] `gb_step` with mode="instruction" works correctly (single-step one CPU instruction).
- [ ] Screenshot output works with MCP's image content type for multimodal display.

## Technical Approach

1. **`src/blobert_mcp/tools/execution.py`** — Execution tools:
   - `gb_step`: Frame mode → loop `pyboy.tick()`. Instruction mode → set hook at next instruction, tick until hit.
   - `gb_run_until`: `pyboy.hook_register(bank, addr, callback)`, tick loop with frame counter for timeout.
   - `gb_get_registers`: Read all from `pyboy.register_file`.

2. **`src/blobert_mcp/tools/input.py`** — Input tools:
   - `gb_press_button`: Map button names to PyBoy API calls (`pyboy.button_press()`/`pyboy.button_release()`).

3. **`src/blobert_mcp/tools/savestate.py`** — Save state tools:
   - `gb_save_state`: `pyboy.save_state(BytesIO())`, store in session dict.
   - `gb_load_state`: `pyboy.load_state(stored_bytesio)`, return post-load registers.
   - State storage: `{state_id: {"name": str, "data": BytesIO, "frame": int, "pc": int, "timestamp": str}}`.

4. **`src/blobert_mcp/tools/visual.py`** — Visual tools:
   - `gb_screenshot`: `pyboy.screen.image` → PIL Image → resize if `scale > 1` → base64 encode.
   - Return as MCP image content block for multimodal display.

5. **Instruction-level stepping** — PyBoy doesn't have native instruction stepping. Approach:
   - Read current PC, decode current instruction to get its size
   - Set hook at `PC + instruction_size`
   - Tick until hook fires
   - Remove hook
   - This has overhead but is correct for single-stepping

6. **Register helper** — Create a `read_all_registers(pyboy)` utility that returns the standardized JSON format, reused by multiple tools.

## Notes

- PyBoy's `tick()` advances one frame (~16.7ms of Game Boy time, ~70224 CPU cycles). This is the fast path.
- Instruction stepping via hooks has overhead. The spec acknowledges this: "instruction-level stepping has overhead due to hook setup/teardown. Frame stepping is fast and preferred for bulk advancement."
- For `gb_run_until`, the timeout is essential to prevent infinite loops. Default 600 frames = ~10 seconds of Game Boy time.
- Screenshot format: PNG is lossless but larger; WebP is smaller. Default to PNG for accuracy.
- `gb_press_button` should handle press-and-release semantics: pressing a button and then stepping a frame is the standard pattern. The tool does NOT auto-step — the LLM controls timing.
- Save states via BytesIO keep everything in memory — no disk I/O. This is fast but means states are lost when the server stops. Acceptable for V1.
