# BLO-050: Multi-ROM session support

## Objective

Enable simultaneous loading and management of multiple Game Boy ROMs, each with its own independent EmulatorSession. This allows comparison workflows such as loading an original ROM alongside a translated or patched version. A SessionManager class coordinates named sessions, and existing tools gain session awareness through an optional `session_id` parameter.

## Context

Currently the MCP server supports a single loaded ROM at a time. For real-world decompilation and translation work, analysts frequently need to compare two or more ROMs side by side — for example, an original Japanese release vs. a fan translation, or a ROM before and after a patch. Multi-session support removes the need to repeatedly unload and reload ROMs, enabling fluid cross-ROM analysis within a single MCP conversation.

## Acceptance Criteria

- [ ] `SessionManager` class exists, holding a dict of `session_id -> EmulatorSession` instances
- [ ] `gb_load_rom` accepts an optional `session_id` parameter (defaults to "default" for backward compatibility)
- [ ] New tool `gb_list_sessions()` returns all active session IDs with ROM metadata (filename, title, size)
- [ ] New tool `gb_switch_session(session_id)` sets the active session for subsequent tool calls
- [ ] New tool `gb_close_session(session_id)` tears down a session and frees resources
- [ ] Existing emulator tools operate on the currently active session transparently
- [ ] Closing a session properly disposes of the PyBoy emulator instance
- [ ] Default session behavior is preserved — existing workflows work without specifying session_id
- [ ] Unit tests cover session lifecycle: create, switch, list, close
- [ ] Edge cases handled: closing active session, switching to nonexistent session, duplicate session IDs

## Technical Approach

1. **SessionManager** (`src/blobert_mcp/domain/session_manager.py` or extend existing emulator module):
   - `SessionManager` dataclass/class with:
     - `sessions: dict[str, EmulatorSession]`
     - `active_session_id: str` (defaults to "default")
   - Methods: `create_session(session_id, rom_path)`, `get_active_session()`, `switch_session(session_id)`, `close_session(session_id)`, `list_sessions()`.
   - Enforce unique session IDs. Return clear errors for invalid operations.

2. **Modify `gb_load_rom` tool**:
   - Add optional `session_id` parameter.
   - Delegate to `SessionManager.create_session()` instead of directly creating an EmulatorSession.
   - If `session_id` already exists, return an error (must close first).

3. **New tools** (`src/blobert_mcp/tools/`):
   - `gb_list_sessions.py` — call `SessionManager.list_sessions()`, format output.
   - `gb_switch_session.py` — call `SessionManager.switch_session(session_id)`, confirm switch.
   - `gb_close_session.py` — call `SessionManager.close_session(session_id)`, confirm teardown.

4. **Update existing emulator tools**:
   - All tools that access the emulator session should go through `SessionManager.get_active_session()` rather than directly holding a session reference.

5. **Testing** (TDD):
   - `tests/domain/test_session_manager.py` — test lifecycle, switching, edge cases.
   - `tests/tools/test_gb_*_session.py` — tool integration tests for new tools.

## Notes

- Memory usage scales linearly with sessions — each PyBoy instance consumes significant RAM. Consider documenting a practical limit (e.g., 3-4 simultaneous sessions).
- The "default" session ID provides full backward compatibility — users who never specify a session_id get the same behavior as before.
- Future enhancement: session-scoped KB databases, so annotations for different ROMs don't collide.
- Consider whether `gb_switch_session` should be implicit (tools accept `session_id` directly) vs. explicit (switch once, all subsequent calls use it). The explicit approach is simpler and chosen here.
