# BLO-071: Comprehensive API reference

## Objective

Create auto-generated documentation of all 25 MCP tools, covering each tool's name, description, parameters (with types and defaults), return format, error codes, and example usage. Include a generation script that introspects FastMCP tool registrations so the documentation stays in sync with the codebase.

## Context

The project has 25 implemented MCP tools but no consolidated API reference. Users and LLM agents interacting with the server need a complete, accurate reference to understand available tools, their parameters, and expected behavior. Manual documentation drifts out of sync — an auto-generated approach ensures accuracy. This is referenced in `docs/mcp-spec.md` as a documentation gap.

## Acceptance Criteria

- [ ] A script (e.g., `scripts/generate_api_docs.py`) introspects FastMCP tool registrations and generates markdown documentation
- [ ] Generated documentation covers all registered MCP tools
- [ ] Each tool entry includes: name, description, parameters (name, type, required/optional, default value), return format, error codes
- [ ] Each tool entry includes at least one example usage (request + response)
- [ ] Output written to `docs/api-reference.md` or `docs/api/` directory with one file per tool category
- [ ] Makefile target `make docs` (or similar) runs the generation script
- [ ] CI check verifies that generated docs are up-to-date (fails if regeneration produces a diff)
- [ ] Generated markdown renders correctly on GitHub

## Technical Approach

- Create a Python script that imports the FastMCP server instance and iterates over registered tools.
- For each tool, extract: function name, docstring, parameter annotations (types, defaults), return type.
- Parse error codes from tool implementations or a centralized error code mapping.
- Format output as markdown with consistent structure per tool.
- Group tools by category (ROM analysis, emulation, knowledge base, etc.) matching the structure in `docs/mcp-spec.md`.
- Add example usage blocks — these can be hand-curated in a separate data file and merged by the script, or extracted from docstrings.
- Add a Makefile target: `make docs` runs the script; `make check-docs` verifies no drift.
- CI integration: add a step that runs `make check-docs` and fails if the generated output differs from committed docs.

## Dependencies

None — can be worked on independently since it documents existing tools.

## Notes

- The generation script should be resilient to tools being added or removed — it should always reflect the current state of registrations.
- Consider whether to output a single `docs/api-reference.md` file or a `docs/api/` directory. A single file is simpler; a directory scales better if the tool count grows significantly.
- Example usage blocks may need to be partially hand-written since auto-generation of realistic examples is difficult. A hybrid approach (auto-generate structure, hand-curate examples) is acceptable.
- The CI check should compare generated output to committed docs and fail with a clear message if they differ.
