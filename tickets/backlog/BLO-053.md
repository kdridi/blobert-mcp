# BLO-053: Watchpoint system

## Objective

Implement `gb_set_watchpoint(address, length?, type?, condition?)` — a tool that sets memory-access breakpoints (watchpoints) that trigger when specified memory locations are read from or written to. This extends the existing breakpoint system (BLO-027) with data-access monitoring.

## Context

The current breakpoint system (BLO-027) supports execution breakpoints (break when PC reaches an address). Watchpoints add data-access breakpoints — essential for reverse engineering to answer questions like "what writes to this RAM address?" or "when does this variable change?". Since PyBoy may not natively support hardware watchpoints, a polling-based approach (snapshot + compare each frame) may be required.

## Acceptance Criteria

- [ ] `gb_set_watchpoint` MCP tool registered and callable
- [ ] Accepts `address` (required) — the memory address to watch
- [ ] Accepts optional `length` parameter (default: 1) — number of bytes to watch starting at address
- [ ] Accepts optional `type` parameter: `"read"`, `"write"`, or `"readwrite"` (default: `"write"`)
- [ ] Accepts optional `condition` parameter for conditional watchpoints (e.g., "value == 0x05")
- [ ] Watchpoint triggers are reported with: address, old value, new value, PC at time of access, frame number
- [ ] Companion tool `gb_list_watchpoints` lists active watchpoints
- [ ] Companion tool `gb_remove_watchpoint` removes a watchpoint by ID
- [ ] Watchpoints coexist with existing execution breakpoints without interference
- [ ] Reuses and extends `domain/breakpoints.py` with watchpoint types
- [ ] Unit tests for domain module written before implementation (TDD)
- [ ] Tool handlers validate inputs and delegate to domain

## Technical Approach

1. **Domain module (extend `domain/breakpoints.py`)**:
   - Add `Watchpoint` dataclass: `id`, `address`, `length`, `type` (read/write/readwrite), `condition`
   - Add `WatchpointHit` dataclass: `watchpoint_id`, `address`, `old_value`, `new_value`, `pc`, `frame`
   - `check_watchpoints(previous_snapshot, current_snapshot, watchpoints) -> list[WatchpointHit]` — compare memory snapshots to detect changes
   - Condition evaluation for conditional watchpoints

2. **Polling-based detection strategy**:
   - Before each frame (or step), snapshot the watched memory regions
   - After execution, compare snapshots to detect writes
   - For read watchpoints: may require instruction-level analysis or accept the PyBoy limitation and document it
   - Trade-off: polling per frame is coarse-grained but compatible with PyBoy's API

3. **Tool handlers (`tools/gb_set_watchpoint.py`, etc.)**:
   - `gb_set_watchpoint`: validate parameters, create watchpoint, register in session state
   - `gb_list_watchpoints`: return all active watchpoints
   - `gb_remove_watchpoint`: remove by ID

4. **Tests (`tests/domain/test_watchpoints.py`)**:
   - Write watchpoint detects value change between snapshots
   - Condition filtering works (only triggers when condition met)
   - Multi-byte watchpoint across address range
   - No false positive when watched region unchanged

## Notes

- **PyBoy limitation**: PyBoy does not expose hardware watchpoints. The polling approach (snapshot + compare) only detects net changes per frame, not individual access events. This means:
  - A write that is immediately overwritten in the same frame may be missed.
  - Read watchpoints cannot be reliably detected without instruction-level tracing.
  - Document these limitations clearly in tool output.
- Consider whether watchpoint state should persist across `gb_step` / `gb_run` calls or be checked inline.
- Future enhancement: if instruction-level stepping becomes available, watchpoints could be made more precise.
