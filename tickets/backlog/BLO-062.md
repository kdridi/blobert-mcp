# BLO-062: Collaborative KB (multi-user)

## Objective

Enable multiple contributors to work on the same ROM's knowledge base by providing tools to merge KB databases, detect and resolve conflicts, and diff between KBs. This unlocks team decompilation workflows where different analysts can work on different parts of a ROM and combine their results.

## Context

Currently, a blobert-mcp KB is a single SQLite database tied to one user's session. For larger ROMs (especially Game Boy Color titles with many banks), splitting the work across multiple contributors is desirable. This requires the ability to merge independently-produced KB databases while handling conflicts (e.g., two analysts labeling the same address differently).

## Acceptance Criteria

- [ ] `kb_merge(other_db_path, strategy?)` tool merges another KB database into the current one
- [ ] `kb_diff(other_db_path)` tool reports differences between two KB databases
- [ ] Merge strategy "theirs_wins" resolves conflicts by preferring the other DB's values
- [ ] Merge strategy "ours_wins" resolves conflicts by preferring the current DB's values
- [ ] Merge strategy "manual" reports conflicts without resolving them, returning a conflict list
- [ ] Non-conflicting entries (different addresses) are merged cleanly regardless of strategy
- [ ] Conflict detection covers: labels, comments, types, and function boundaries at the same address
- [ ] Merge produces a summary report (entries added, entries updated, conflicts found/resolved)
- [ ] Diff output shows added, removed, and changed entries between two KBs
- [ ] Integration with BLO-039 export and BLO-040 import for file-based KB exchange
- [ ] Tests cover all three merge strategies with synthetic KB data

## Technical Approach

- **kb_diff**: Open both SQLite databases, iterate over all annotation tables (labels, comments, types, functions), and compare entries by address. Output categorized as: added (in other, not in ours), removed (in ours, not in other), changed (same address, different values).
- **kb_merge**: Use kb_diff as the foundation. For each difference:
  - Added entries: insert into current DB.
  - Changed entries: apply based on strategy (theirs_wins, ours_wins, or collect for manual review).
  - Removed entries: optionally flag but do not delete (conservative merge).
- **MCP tools**: Register `kb_merge` and `kb_diff` as MCP tools. `kb_merge` takes `other_db_path` and optional `strategy` parameter (default: "manual"). `kb_diff` takes `other_db_path`.
- **Domain layer**: Core merge/diff logic lives in `domain/` as pure functions operating on data extracted from both databases. Tool handlers in `tools/` manage DB connections and format output.
- **Conflict representation**: A conflict is a dataclass with `address`, `field`, `ours_value`, `theirs_value`. The manual strategy returns a list of these.

## Notes

- SQLite's single-writer model means merges must be done offline (not concurrent writes). This is acceptable for the initial implementation.
- Future enhancement: version vectors or timestamps per-entry to enable smarter merge (last-write-wins with causality tracking).
- Consider whether "removed" entries in a diff should be deletable via merge or only flagged. Conservative approach: never auto-delete.
- The file-based exchange workflow: analyst A exports KB (BLO-039), sends file to analyst B, analyst B runs `kb_merge` to incorporate.
- Git-based workflow is also possible: each analyst commits their KB SQLite file, and merge is run after pulling.
