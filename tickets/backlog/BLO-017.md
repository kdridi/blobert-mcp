# BLO-017: Disassembly label enrichment from knowledge base

## Objective

Wire the knowledge base label lookup into the disassembly output so that instructions referencing annotated addresses display their human-readable labels as comments (e.g., `CALL 0x1234 ; play_sound`). After this ticket, every disassembly output is automatically enriched with the LLM's accumulated knowledge base annotations.

## Context

This is the integration glue between the disassembler (BLO-016) and the knowledge base (BLO-005). It was intentionally split out of BLO-003 because it has a hard dependency on both — neither can be started without the other being complete. The disassembler was designed with a `label_resolver` parameter stub precisely to make this integration clean when the time comes.

Covers the label enrichment aspect of spec §1 (disassembler output) and the KB integration point described in spec §8.

## Acceptance Criteria

- [ ] `disassemble_range`, `disassemble_function`, `disassemble_at_pc` in `domain/disasm/disassembler.py` each accept an optional `label_resolver: Callable[[int], str | None]` parameter. When an instruction operand is an absolute address and `label_resolver(address)` returns a non-None string, the operand is rendered as `"0x1234 ; label"`.
- [ ] MCP tool wrappers in `tools/disasm.py` pass `session.kb.get_label` as `label_resolver` when `session.kb` is active; pass `None` (no-op) when no KB is loaded. The tools are transparent to callers — no new parameters needed at the MCP level.
- [ ] Tests verify: label injected into `CALL`, `JP`, `LD A,(a16)` and similar address-referencing operands; correct no-op when `label_resolver=None`; correct no-op when `label_resolver` returns `None` for an address.

## Technical Approach

1. **Update `src/blobert_mcp/domain/disasm/disassembler.py`**:
   - Add `label_resolver: Callable[[int], str | None] | None = None` parameter to all three functions
   - After resolving each operand, if it is an absolute address and `label_resolver` is not None, call it and append `; <label>` to the operand string if a label is returned
   - Domain layer still has no project imports — `label_resolver` is passed in from outside

2. **Update `src/blobert_mcp/tools/disasm.py`**:
   - Construct `label_resolver = session.kb.get_label if session.kb else None`
   - Pass `label_resolver` to the three domain functions

3. **Update tests**:
   - `tests/domain/disasm/test_disassembler.py` — add test cases with a mock `label_resolver` lambda
   - `tests/tools/test_disasm.py` — add test cases verifying KB wiring when session.kb is set and when it is None

## Dependencies

- BLO-016: Disassembly algorithms and MCP tools (the `label_resolver` parameter stub)
- BLO-005: Knowledge base with SQLite storage (the `KnowledgeBase.get_label()` method)

## Notes

- The `label_resolver` callable is the only interface between the disassembler and the KB. No direct KB import in the domain layer — ever.
- Address-referencing operand types to enrich: `a16` (absolute), `(a16)`, resolved JR targets (r8 → absolute). Do not enrich register names, immediate data values, or relative offsets before resolution.
- `KnowledgeBase.get_label(address, bank=None)` is defined in BLO-005 as a fast lookup method. BLO-017 will use it.
