# BLO-059: Performance benchmark suite

## Objective

Create a performance benchmark suite that measures and tracks the speed of critical operations (disassembly, KB queries, ROM scanning, pattern search). Results are output as JSON for CI tracking. Done when benchmarks run via `make bench` and produce consistent, comparable timing data.

## Context

With 25 MCP tools and a growing KB, there is no visibility into performance characteristics or regressions. A benchmark suite establishes baselines for critical paths and enables tracking over time. This is especially important as the KB grows — query latency and full-ROM operations could degrade without early detection. Depends on KB infrastructure (BLO-005) and KB queries (BLO-016).

## Acceptance Criteria

- [ ] Benchmark suite in `benchmarks/` directory
- [ ] Benchmark: disassembly throughput (instructions decoded per second)
- [ ] Benchmark: KB query latency (annotation lookup, function search)
- [ ] Benchmark: full ROM scan time (complete disassembly pass)
- [ ] Benchmark: pattern search speed (byte pattern matching across ROM data)
- [ ] JSON output with timing results (mean, median, min, max, iterations)
- [ ] `make bench` Makefile target runs the full benchmark suite
- [ ] Benchmarks use synthetic data (no copyrighted ROMs)
- [ ] Tests for benchmark harness itself (validates output format)

## Technical Approach

1. **Benchmark harness** (`benchmarks/`):
   - `conftest.py` — shared fixtures (synthetic ROM data, pre-populated KB)
   - `bench_disassembly.py` — disassembly throughput benchmarks
   - `bench_kb.py` — KB query latency benchmarks
   - `bench_scan.py` — full ROM scan benchmarks
   - `bench_pattern.py` — pattern search benchmarks

2. **Implementation options** (choose one):
   - **pytest-benchmark**: Integrates with existing pytest infrastructure, provides statistical analysis, JSON export via `--benchmark-json`
   - **Custom harness**: Lightweight `timeit`-based runner with JSON output; fewer dependencies but less statistical rigor
   - pytest-benchmark is preferred if it integrates cleanly

3. **Synthetic test data**:
   - Generate byte arrays of varying sizes (1KB, 32KB, 256KB, 1MB) for throughput scaling
   - Pre-populate SQLite KB with synthetic annotations (100, 1000, 10000 entries) for query benchmarks

4. **Makefile integration**:
   - `make bench` target runs `uv run pytest benchmarks/ --benchmark-json=benchmarks/results.json` (or equivalent)
   - Results file `.gitignore`'d (machine-specific)

5. **JSON output schema**:
   - Each benchmark entry: `{"name": "...", "mean": float, "median": float, "min": float, "max": float, "rounds": int}`
   - Top-level metadata: timestamp, Python version, platform

## Notes

- pytest-benchmark can be added as a dev dependency via `uv add --dev pytest-benchmark`
- Benchmark results are inherently machine-dependent; CI tracking should compare relative changes, not absolute values
- Consider adding a `--benchmark-compare` step in CI to flag regressions above a threshold (e.g., >20% slower)
- Keep synthetic data generation deterministic (seeded random) for reproducibility
