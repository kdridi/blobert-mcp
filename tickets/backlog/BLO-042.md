# BLO-042: KB annotation history

## Objective

Implement the `kb_get_history` tool that returns the modification history of an annotation at a given address. This requires a new `annotation_history` SQLite table with triggers that automatically record changes when annotations are updated. This ticket is done when history is recorded automatically on annotation changes and the tool returns chronological history entries.

## Context

The spec (section 8) defines `kb_get_history` (P2) for tracking how annotations evolve over time. This is valuable for understanding the decompilation process -- seeing what labels/comments were revised and why. The existing KB (BLO-005) stores current annotation state but does not track changes. A new history table and SQLite triggers are needed.

See `docs/mcp-spec.md` section 8 for the tool definition.

## Acceptance Criteria

- [ ] New `annotation_history` SQLite table exists with columns: id, address, bank, timestamp, old_value, new_value, field_changed, reason
- [ ] SQLite triggers automatically insert history records when annotations are updated or deleted
- [ ] `kb_get_history(address, bank?)` tool is registered and callable via MCP
- [ ] `kb_get_history` returns chronological list of modifications (timestamp, old_value, new_value, reason)
- [ ] `kb_get_history` filters by bank when provided
- [ ] `kb_get_history` returns an empty list (not an error) when no history exists for an address
- [ ] Domain module handles history formatting (pure functions, no tool/MCP imports)
- [ ] Unit tests exist for domain logic (TDD)
- [ ] Tool handler tests exist
- [ ] KB schema migration or initialization creates the history table and triggers

## Technical Approach

1. **Schema changes**: Add `annotation_history` table to the KB SQLite schema. Create `AFTER UPDATE` and `AFTER DELETE` triggers on the annotations table that insert records into `annotation_history` capturing the old values, new values, changed field, and timestamp.
2. **Domain module**: Create history formatting functions in a domain module (e.g., `domain/kb_history.py`) that query the history table and return structured results. Pure functions that accept a DB connection.
3. **Tool handler**: Thin wrapper that validates address/bank input, calls domain function, and formats the chronological output.
4. **Migration**: Update the KB initialization code to include the new table and triggers. Ensure backward compatibility -- existing databases without the history table should have it added on next initialization.
5. **Tests**: Write domain tests first (TDD). Test trigger behavior by performing annotation updates and verifying history records are created.

## Notes

- The `reason` field in history entries can be populated if the annotation update API is extended to accept a reason parameter. Initially, it may be null for trigger-generated entries.
- SQLite triggers fire within the same transaction, so history recording adds minimal overhead.
- Consider whether to track history for functions and variables tables as well (future scope -- not required for this ticket).
