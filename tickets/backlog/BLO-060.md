# BLO-060: Web dashboard / viewer

## Objective

Build a web-based UI that visualizes ROM analysis results stored in the KB. The dashboard provides a ROM memory map viewer, disassembly browser, KB annotation overlay, live screenshot display, and VRAM tile viewer. It runs as a separate entry point from the MCP server and reads from the KB database in read-only mode.

## Context

The current workflow is entirely CLI/MCP-driven. A visual dashboard would make it significantly easier to navigate large ROMs, inspect disassembly with annotations, and correlate emulator screenshots with memory regions. This is a large feature that will likely need to be broken into sub-tickets once the initial architecture is established.

## Acceptance Criteria

- [ ] Separate entry point (e.g., `blobert-web` or `python -m blobert_mcp.web`) launches the dashboard
- [ ] ROM memory map viewer displays address ranges with color-coded regions (code, data, unknown)
- [ ] Disassembly browser shows decoded instructions with navigation (jump to address, scroll)
- [ ] KB annotation overlay displays labels, comments, and types from the knowledge base inline with disassembly
- [ ] Live screenshot view renders the current emulator screen (via `gb_screenshot` PIL images)
- [ ] VRAM tile viewer displays tile data extracted from VRAM
- [ ] Dashboard reads from KB database directly (read-only, no writes)
- [ ] Lightweight Python web framework used (Flask or FastAPI)
- [ ] Frontend uses HTMX or simple JS (no heavy SPA framework)
- [ ] Documentation covers how to start and use the dashboard

## Technical Approach

- Use Flask or FastAPI as the backend web framework.
- Serve a minimal frontend using HTMX for dynamic updates or vanilla JS for interactivity.
- Create a read-only database connection to the KB SQLite database.
- Implement API endpoints for each viewer component (memory map, disassembly, annotations, screenshot, VRAM tiles).
- The ROM memory map can reuse domain logic from existing MCP tools.
- Screenshots can be served as base64-encoded PNGs or streamed as image responses.
- VRAM tile viewer renders tile data as a grid of small images.
- Entry point should be a separate CLI command or module invocation, not coupled to the MCP server.
- This is a large undertaking — plan to split into sub-tickets for individual viewer components.

## Notes

- This is a large feature. Consider creating sub-tickets for: (1) project scaffolding + entry point, (2) memory map viewer, (3) disassembly browser, (4) KB annotation overlay, (5) live screenshot, (6) VRAM tile viewer.
- Flask is simpler for a read-only dashboard; FastAPI offers async and auto-generated API docs.
- HTMX keeps the frontend lightweight while enabling dynamic content updates without a full JS framework.
- Security is minimal concern since this is a local development tool, but binding to localhost only is recommended.
