# BLO-054: Plugin/hook system for custom analysis

## Objective

Design and implement a plugin/hook architecture that allows user-defined analysis plugins to extend the server's capabilities without modifying core code. Definition of done: plugins can be discovered, loaded, enabled/disabled, and their hooks are invoked at the correct execution points during ROM analysis.

## Context

As the decompilation tooling matures, users and community contributors will want to add custom analysis passes (e.g., pattern detectors, annotation generators, heuristic labelers) without forking or patching the core server. A plugin system provides a stable extension point, enabling a plugin ecosystem around blobert-mcp.

## Acceptance Criteria

- [ ] Plugin interface defined: Python modules that register hooks (`on_instruction`, `on_frame`, `on_memory_write`, etc.)
- [ ] Plugin discovery: server scans a `plugins/` directory and auto-loads conforming modules
- [ ] Hook dispatch: registered hooks are called at the appropriate execution points with correct context
- [ ] MCP tool `gb_list_plugins()` returns all discovered plugins with their enabled/disabled status
- [ ] MCP tool `gb_enable_plugin(name)` enables a plugin by name
- [ ] MCP tool `gb_disable_plugin(name)` disables a plugin by name
- [ ] Disabled plugins' hooks are not invoked
- [ ] Plugin errors are isolated — a failing plugin does not crash the server or other plugins
- [ ] Domain logic for plugin lifecycle is covered by tests (TDD)
- [ ] Architecture documented in `docs/architecture.md` or a dedicated `docs/plugins.md`

## Technical Approach

1. **Domain layer** (`src/blobert_mcp/domain/plugins/`):
   - Define a `Plugin` dataclass: name, version, description, enabled flag, hook registry.
   - Define hook protocol/ABC: `PluginHook` with methods like `on_instruction(ctx)`, `on_frame(ctx)`, `on_memory_write(ctx)`.
   - `PluginManager` class: discover, load, enable/disable, dispatch hooks.
   - Plugin errors caught and logged, never propagated to caller.

2. **Plugin discovery**:
   - Scan `plugins/` directory for Python modules/packages containing a `register(manager)` entry point.
   - Use `importlib` for dynamic loading.

3. **Tool layer** (`src/blobert_mcp/tools/`):
   - `gb_list_plugins()` — delegates to `PluginManager.list_plugins()`.
   - `gb_enable_plugin(name)` — delegates to `PluginManager.enable(name)`.
   - `gb_disable_plugin(name)` — delegates to `PluginManager.disable(name)`.

4. **Integration**: Wire hook dispatch into existing execution/analysis code paths (depends on BLO-013 execution engine).

## Dependencies

- BLO-013: Execution engine must exist so hooks have execution points to attach to.
