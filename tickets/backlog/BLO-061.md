# BLO-061: Ghidra/IDA bridge

## Objective

Create bidirectional integration between the blobert-mcp knowledge base and the two major reverse engineering platforms: Ghidra and IDA Pro. This enables a hybrid workflow where LLM-assisted annotation in blobert-mcp can be combined with traditional RE tool analysis.

## Context

Professional ROM reverse engineering often involves Ghidra or IDA Pro for static analysis. Currently, annotations and labels created via blobert-mcp MCP tools are siloed in the KB. Bridging these tools allows analysts to export blobert-mcp KB data into Ghidra/IDA projects and import analysis results back, creating a feedback loop between LLM-assisted and manual decompilation.

## Acceptance Criteria

- [ ] Export: KB data can be exported to a Ghidra XML project file
- [ ] Export: KB data can be exported to an IDA `.idc` script
- [ ] Import: Ghidra analysis results (labels, comments, types) can be imported into the KB
- [ ] Import: IDA analysis results (labels, comments, types) can be imported into the KB
- [ ] GhidraScript: Python plugin that calls blobert-mcp MCP tools for in-Ghidra integration
- [ ] IDAPython: Script for synchronizing IDA annotations with the KB
- [ ] Export leverages BLO-039 export infrastructure
- [ ] Import leverages BLO-040 import infrastructure
- [ ] Round-trip test: export from KB, import into tool, modify, export from tool, import back to KB
- [ ] Documentation covers setup and usage for both Ghidra and IDA workflows

## Technical Approach

- **Export to Ghidra**: Generate Ghidra XML project format containing labels, comments, and memory regions from the KB. Use BLO-039 export as the data source.
- **Export to IDA**: Generate `.idc` scripts that apply KB labels, comments, and type annotations. IDC is IDA's scripting format and is widely compatible.
- **Import from Ghidra**: Parse Ghidra XML export or use Ghidra's flat API output to extract labels, comments, and function boundaries. Feed into BLO-040 import pipeline.
- **Import from IDA**: Parse IDA `.idc` or `.lst` files to extract analysis results. Alternatively, use IDAPython to export JSON and import that.
- **GhidraScript plugin**: A Python script for Ghidra's Jython environment that connects to blobert-mcp (via subprocess or HTTP if a web API exists) to query/update the KB.
- **IDAPython script**: A Python script for IDA that synchronizes annotations bidirectionally.
- Game Boy-specific considerations: SM83 processor module in Ghidra, banked memory model translation.

## Dependencies

- BLO-039: KB export infrastructure (provides the data extraction layer)
- BLO-040: KB import infrastructure (provides the data ingestion layer)

## Notes

- Ghidra has a Game Boy loader (GhidraBoy) that may simplify processor/memory mapping.
- IDA requires a Game Boy processor module; ensure address translation handles ROM banking correctly.
- The GhidraScript plugin runs in Jython (Python 2.7 in older Ghidra, Python 3 in Ghidra 11+). Target Ghidra 11+ for Python 3 compatibility.
- Consider whether the bridge should work via MCP tool calls, direct DB access, or file-based exchange. File-based is simplest and most portable.
- This is a large feature. Consider sub-tickets for: (1) Ghidra export, (2) Ghidra import, (3) IDA export, (4) IDA import, (5) GhidraScript plugin, (6) IDAPython script.
