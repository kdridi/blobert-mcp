# BLO-072: v1.0 release preparation

## Objective

Prepare the blobert-mcp project for its first stable release (v1.0.0). This includes updating the version number, finalizing the changelog, creating a GitHub release with release notes, verifying PyPI publishing, and ensuring all release criteria are met before tagging.

## Context

With 25 MCP tools implemented and documentation nearing completion, the project is approaching a state where a stable v1.0 release is appropriate. A formal release establishes a baseline for users, enables semantic versioning for future changes, and signals that the project is ready for broader adoption. This ticket depends on documentation tickets (BLO-070, BLO-071) and stability work (BLO-066) being completed first.

## Acceptance Criteria

- [ ] Version updated to `1.0.0` in `pyproject.toml`
- [ ] `CHANGELOG.md` finalized with all changes since project inception, organized by category
- [ ] All P0 and P1 tools implemented and passing tests
- [ ] Test coverage exceeds 90% (measured via `pytest --cov`)
- [ ] No known critical bugs (all open bug tickets are P2 or lower)
- [ ] Documentation complete: README (BLO-070), API reference (BLO-071), architecture, testing strategy
- [ ] GitHub release created with tag `v1.0.0` and detailed release notes
- [ ] PyPI publish verified (package installs correctly via `pip install blobert-mcp`)
- [ ] README badges updated to reflect published version and passing CI
- [ ] Release announcement drafted (can be GitHub release notes or a separate document)

## Technical Approach

- **Version bump**: Update the `version` field in `pyproject.toml` to `1.0.0`. Follow semantic versioning conventions.
- **Changelog**: Review all commits since project start. Organize entries into: Added, Changed, Fixed, Removed. Follow Keep a Changelog format.
- **Release criteria check**: Run the full test suite (`make test`), check coverage (`make coverage` or `pytest --cov`), review open tickets for critical bugs.
- **GitHub release**: Use `gh release create v1.0.0` with release notes summarizing key features, installation instructions, and known limitations.
- **PyPI publish**: Ensure `pyproject.toml` has correct metadata (name, description, classifiers, URLs). Build with `uv build` or `python -m build`. Publish with `twine upload` or GitHub Actions workflow.
- **Badge update**: After PyPI publish, add/update shields.io badges in README for PyPI version.
- **Post-release**: Create a `v1.0.x` maintenance branch if needed, or continue development on `main` toward v1.1.

## Notes

- Consider whether to publish to PyPI immediately or first do a test publish to TestPyPI.
- The GitHub Actions workflow for publishing may need to be created or updated as part of this ticket.

- A pre-release checklist (manual or automated) would help ensure nothing is missed on release day.
