# BLO-002: ROM loading, session management & static reading tools

## Objective

Implement the P0 tools that load a ROM, report session state, and read ROM/memory data without executing. After this ticket, an LLM can load a Game Boy ROM, inspect its header, read raw bytes from any bank, and understand the memory layout — the first 3 steps of the example workflow.

## Context

These tools form the data-access layer. They are read-only (no execution, no state mutation beyond loading) and are prerequisites for both the disassembler (BLO-003) and dynamic execution (BLO-004).

Covers spec sections: §1 (Static analysis — P0 tools), §7 (Hardware — P0 tools), §9 (Session — P0 tools).

## Acceptance Criteria

- [ ] `gb_load_rom` — Loads a ROM file into PyBoy, returns status + parsed header. Validates file exists. Supports `headless` parameter.
- [ ] `get_rom_header` — Parses bytes at 0x0100-0x014F, returns JSON with title, MBC type, ROM/RAM size, licensee, CGB/SGB flags, checksums.
- [ ] `get_memory_map` — Returns the static Game Boy memory map (ROM0, ROMX, VRAM, SRAM, WRAM, OAM, I/O, HRAM) with address ranges and access types.
- [ ] `read_rom_bytes` — Reads up to 4096 bytes from ROM at a given address/bank, returns hexdump + ASCII.
- [ ] `get_vector_table` — Reads RST vectors (0x00-0x38), interrupt vectors (0x40-0x60), and entry point (0x100), returns addresses with first few bytes at each.
- [ ] `gb_get_bank_info` — Returns MBC type, current ROM/RAM bank, total bank count.
- [ ] `gb_read_banked` — Reads memory from a specific bank using `pyboy.memory[bank, address]`.
- [ ] `gb_read_memory` — Reads up to 4096 bytes from any memory address (RAM, VRAM, I/O), returns hexdump + ASCII.
- [ ] `gb_get_interrupt_status` — Returns IE (0xFFFF), IF (0xFF0F) with individual flag breakdown.
- [ ] `get_session_info` — Returns ROM loaded status, ROM title, frame count, PC, annotation count, save state count.
- [ ] `gb_reset` — Resets the emulator without reloading the ROM.
- [ ] All tools return proper error responses when no ROM is loaded.
- [ ] Hexdump output format is consistent: offset, hex bytes (space-separated, 16 per line), ASCII sidebar.

## Technical Approach

1. **`src/blobert_mcp/tools/static.py`** — Static analysis tools: `get_rom_header`, `get_memory_map`, `read_rom_bytes`, `get_vector_table`.

2. **`src/blobert_mcp/tools/session.py`** — Session tools: `gb_load_rom`, `get_session_info`, `gb_reset`.

3. **`src/blobert_mcp/tools/memory.py`** — Memory reading tools: `gb_read_memory`, `gb_read_banked`, `gb_get_bank_info`, `gb_get_interrupt_status`.

4. **`src/blobert_mcp/utils/hexdump.py`** — Shared hexdump formatter (address + hex + ASCII).

5. **ROM header parsing** — Decode bytes per the Game Boy header spec:
   - 0x0134-0x0143: Title
   - 0x0147: Cartridge type (MBC)
   - 0x0148: ROM size
   - 0x0149: RAM size
   - 0x014B: Old licensee
   - 0x0143: CGB flag
   - 0x0146: SGB flag
   - 0x014D: Header checksum
   - 0x014E-0x014F: Global checksum

6. **Register tools in `server.py`** — Import and register each tool group.

## Notes

- The `get_memory_map` tool returns static data — no emulator needed. It should work even when no ROM is loaded.
- `gb_get_bank_info` needs MBC-type-specific logic to determine the current bank. Start with MBC1/MBC3/MBC5 which cover >95% of Game Boy ROMs.
- Error handling pattern: every tool should check `session.pyboy is not None` before accessing the emulator. Return `{"error": "NO_ROM_LOADED", "message": "..."}` if not.
