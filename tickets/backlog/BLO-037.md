# BLO-037: Execution tracing and profiling tools

## Objective

Implement four tracing and profiling MCP tools that use PyBoy's `hook_register` API to instrument execution: `gb_trace_start`, `gb_trace_stop`, `gb_get_hotspots`, and `gb_trace_bank_switches`. Done when all four tools are registered, functional, and tested, enabling the LLM to record execution traces, identify hot code paths, and log ROM bank switch activity.

## Context

Tracing and profiling are P1 tools defined in `docs/mcp-spec.md` sections 3 and 7. They build on the EmulatorSession and hook infrastructure completed in BLO-021. PyBoy does not provide native instruction-level tracing, so these tools use `hook_register()` for targeted instrumentation and PC sampling for statistical profiling.

From the spec:
- `gb_trace_start` (section 3): Hook addresses in a range, record executed instructions. Feasible for small ranges (single functions).
- `gb_trace_stop` (section 3): Stop recording and return the trace data.
- `gb_get_hotspots` (section 3): Sample PC each tick over a duration, report most-executed addresses.
- `gb_trace_bank_switches` (section 7): Hook MBC register writes (0x2000-0x3FFF), log bank switch events with frame/PC/value.

## Acceptance Criteria

- [ ] Domain module `domain/trace.py` exists with dataclasses for trace records, hotspot entries, and bank switch events
- [ ] `gb_trace_start(address, length, format?, max_instructions?)` registers hooks across the specified address range and begins recording
- [ ] `gb_trace_stop(trace_id)` removes hooks and returns the recorded trace data in the requested format
- [ ] Minimal format returns address + opcode; verbose format returns address + opcode + full register snapshot
- [ ] `gb_get_hotspots(duration_frames, top_n?)` samples PC after each tick for the specified duration and returns the top N addresses by execution count
- [ ] `gb_trace_bank_switches(duration_frames)` hooks MBC register writes and logs each bank switch with frame number, PC, new bank value, and write address
- [ ] Trace recording respects `max_instructions` limit (default 1000, max 10000) and stops collecting when reached
- [ ] All four tools are registered as MCP tools with proper input validation and error handling
- [ ] Unit tests for `domain/trace.py` data structures and logic
- [ ] Tool-level tests for all four MCP handlers

## Technical Approach

1. **Domain module `domain/trace.py`:**
   - `TraceRecord` dataclass: `address`, `opcode`, `mnemonic`, `registers` (optional, for verbose mode).
   - `TraceSession` dataclass: `trace_id`, `format`, `max_instructions`, `records: list[TraceRecord]`, `active: bool`.
   - `HotspotEntry` dataclass: `address`, `count`, `percentage`.
   - `BankSwitchEvent` dataclass: `frame`, `pc`, `new_bank`, `write_address`.
   - Pure functions for aggregating hotspot data from PC samples and sorting by count.

2. **Trace start/stop implementation:**
   - `gb_trace_start`: Disassemble the specified range to find instruction boundaries. Register a hook at each instruction address. Each hook callback appends a `TraceRecord` to the active `TraceSession`. Return a `trace_id`.
   - `gb_trace_stop`: Deregister all hooks for the given `trace_id`. Return the collected records.
   - Store active traces in the EmulatorSession (or a dedicated trace manager).

3. **Hotspot profiling:**
   - `gb_get_hotspots`: Run `pyboy.tick()` for `duration_frames` frames. After each tick, read `register_file.PC` and increment a counter dict. Sort by count descending, return top N with percentage of total samples.

4. **Bank switch logging:**
   - `gb_trace_bank_switches`: Register hooks at MBC write addresses (0x2000-0x3FFF for ROM bank select on most MBC types). Each hook callback logs `BankSwitchEvent` with current frame, PC, and written value. Run for `duration_frames`, then remove hooks and return events.

5. **Testing (TDD):**
   - Write tests for `domain/trace.py` dataclass construction and hotspot aggregation logic using synthetic data.
   - Tool-level tests mock the EmulatorSession and hook API to verify correct hook registration/deregistration and data collection.

## Notes

- Hooking every instruction in a large range is expensive. The spec recommends keeping ranges small (100-500 bytes, i.e., single functions). Consider warning or refusing if the range exceeds a threshold.
- PC sampling in `gb_get_hotspots` gives statistical approximation, not exact execution counts. This is a known limitation documented in the spec.
- Bank switch detection depends on MBC type. MBC1 uses 0x2000-0x3FFF for ROM bank, MBC3/MBC5 use similar ranges but with different bit widths. The implementation should consult the ROM header to determine the correct MBC register addresses.
- Trace IDs should be unique per session and managed so that orphaned traces (started but never stopped) are cleaned up on ROM reload or session reset.
