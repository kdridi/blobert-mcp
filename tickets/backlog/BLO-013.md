# BLO-013: Session & static reading tools

## Objective

Implement the first tool layer: session management (load ROM, query state, reset) and static reading tools (ROM header, memory map, raw bytes, vector table). After this ticket, 7 MCP tools are registered and functional — an LLM can load a Game Boy ROM, inspect its header, browse the memory map, read raw bytes from any bank, and view the vector table.

## Context

This is the second of three tickets decomposed from BLO-002. It builds the `tools/session.py` and `tools/static.py` modules on top of the domain layer completed in BLO-012. These are the foundational tools that BLO-003 (disassembler) and BLO-004 (execution) depend on.

Covers spec sections: §1 (Static analysis — P0 tools), §9 (Session — P0 tools).

See `docs/architecture.md` for the tool implementation pattern (5-step recipe).

## Acceptance Criteria

- [ ] `tools/` and `tests/tools/` directories exist with `__init__.py`.
- [ ] `gb_load_rom` — Loads a ROM file via EmulatorSession, returns status + parsed header (using `domain/rom_header`). Validates file exists. Supports `headless` parameter (default true). Returns `{"error": "FILE_NOT_FOUND", ...}` if path invalid.
- [ ] `get_session_info` — Returns `rom_loaded`, `rom_title`, `frame_count`, `pc`, `annotation_count`, `save_state_count`. Works when no ROM loaded (returns `rom_loaded: false` with null fields).
- [ ] `gb_reset` — Resets the emulator without reloading the ROM. Returns confirmation. Returns `{"error": "NO_ROM_LOADED", ...}` if no ROM loaded.
- [ ] `get_rom_header` — Reads bytes 0x0100-0x014F from emulator memory, delegates to `domain/rom_header.parse()`, returns structured JSON. Returns error if no ROM loaded.
- [ ] `get_memory_map` — Returns the static Game Boy memory map from `domain/memory_map`. Works even when no ROM is loaded (static data).
- [ ] `read_rom_bytes` — Reads up to 4096 bytes from ROM at `address`/`bank`, delegates to `utils/hexdump` for formatting. Validates length (1-4096) and address range. Returns error if no ROM loaded.
- [ ] `get_vector_table` — Reads RST vectors, interrupt vectors, and entry point from ROM using `domain/vectors` definitions. Returns addresses with first bytes at each vector. Returns error if no ROM loaded.
- [ ] All 7 tools are registered in `server.py` and accessible via MCP. Integration tests cover happy path and error paths (no ROM loaded).

## Technical Approach

### Tool implementation pattern (from `docs/architecture.md`)

Every tool follows the 5-step recipe:
1. **Validate inputs** — check types, ranges, limits
2. **Check preconditions** — is ROM loaded?
3. **Read raw bytes** — `session.pyboy.memory[...]`
4. **Delegate to domain** — pass bytes to pure function
5. **Format and return** — use `utils/hexdump` if needed

### Files

**`src/blobert_mcp/tools/session.py`**
- `gb_load_rom(rom_path: str, headless: bool = True) -> dict`
- `get_session_info() -> dict`
- `gb_reset() -> dict`

**`src/blobert_mcp/tools/static.py`**
- `get_rom_header() -> dict`
- `get_memory_map() -> dict`
- `read_rom_bytes(address: int, length: int = 256, bank: int | None = None) -> dict`
- `get_vector_table() -> dict`

**`src/blobert_mcp/server.py`** — Import and register both tool modules.

### Error handling

All tools that require a loaded ROM check `session.rom_loaded` first:
```python
if not session.rom_loaded:
    return {"error": "NO_ROM_LOADED", "message": "Load a ROM first with gb_load_rom."}
```

### Integration tests

Tests use a minimal test ROM (generated programmatically or stored in `tests/fixtures/`) or a mock EmulatorSession that returns predetermined bytes.
