# BLO-022: Instruction stepping + screenshot

## Objective

Add instruction-level stepping to `gb_step` and implement the `gb_screenshot` tool. After this ticket, an LLM can single-step individual CPU instructions (for precise analysis) and capture the Game Boy screen as an image (for visual context). This completes all P0 tools from BLO-004.

## Context

This is the third and final ticket decomposed from BLO-004 (rejected). It adds the two remaining capabilities that require distinct technical approaches: hook-based instruction stepping (using the existing SM83 decoder) and image capture with MCP multimodal output.

Covers spec sections: §2 (`gb_step` instruction mode), §6 (`gb_screenshot` — P0).

## Acceptance Criteria

- [ ] `gb_step` with mode="instruction" — Advances execution by `count` individual CPU instructions. Uses hook-based stepping: decode current instruction to get size, set hook at PC + size, tick until hook fires, remove hook.
- [ ] Instruction size decoded via existing `domain/disasm/decoder.decode_instruction()`.
- [ ] `gb_step` instruction mode works correctly for all instruction sizes (1-byte NOP through 3-byte JP a16).
- [ ] `gb_step` instruction mode handles CB-prefixed instructions (2 bytes for prefix + opcode).
- [ ] `gb_screenshot` — Captures the screen as a base64-encoded image. Supports `format` parameter ("png" | "webp", default "png") and `scale` parameter (int, default 1) for upscaling.
- [ ] Screenshot output uses MCP's image content type for multimodal display.
- [ ] Both tools return proper errors when no ROM is loaded.
- [ ] Integration tests cover instruction stepping (various instruction sizes) and screenshot output.

## Technical Approach

### Instruction stepping (`tools/execution.py` enhancement)

Update the existing `gb_step` tool to handle mode="instruction":

```python
# For each instruction step:
# 1. Read current PC
pc = session.pyboy.register_file.PC
# 2. Read bytes at PC (up to 3 bytes — max instruction size)
raw = bytes(session.pyboy.memory[pc:pc + 3])
# 3. Decode to get instruction size
instr = decoder.decode_instruction(raw, pc)
next_pc = (pc + instr.size) & 0xFFFF
# 4. Set hook at next_pc
hit = False
def on_hit(context): nonlocal hit; hit = True
session.pyboy.hook_register(-1, next_pc, on_hit, None)
# 5. Tick until hook fires
while not hit: session.pyboy.tick()
# 6. Remove hook
session.pyboy.hook_deregister(next_pc)
```

**Edge cases:**
- CB-prefixed instructions: `decoder.decode_instruction` already handles these (returns size=2)
- Jumps/calls: the hook is set at PC + size (the *sequential* next instruction). If the instruction jumps elsewhere, the hook at the old next_pc will never fire in that step. Need timeout protection.
- Memory bank boundaries: use bank=-1 (current bank) for hooks.

### Screenshot (`tools/visual.py`)

**`src/blobert_mcp/tools/visual.py`** — `register_visual_tools(mcp, session)`:

```python
import base64
from io import BytesIO

@mcp.tool()
def gb_screenshot(format: str = "png", scale: int = 1) -> dict:
    image = session.pyboy.screen.image
    if scale > 1:
        image = image.resize((image.width * scale, image.height * scale), Image.NEAREST)
    buf = BytesIO()
    image.save(buf, format=format.upper())
    data = base64.b64encode(buf.getvalue()).decode("ascii")
    # Return MCP image content type
    media_type = "image/png" if format == "png" else "image/webp"
    return {"type": "image", "data": data, "mimeType": media_type}
```

## Notes

- PyBoy's `hook_register(bank, addr, callback, context)` is the mechanism for instruction stepping. Bank -1 means any bank.
- The spec acknowledges instruction stepping has overhead: "instruction-level stepping has overhead due to hook setup/teardown. Frame stepping is fast and preferred for bulk advancement."
- For jumps/calls, the hook at sequential next_pc won't fire. The stepping loop needs a frame-based timeout to avoid hanging. If the instruction is a jump, we may need to detect this and set the hook at the jump target instead.
- `decoder.decode_instruction()` already resolves relative jumps to absolute addresses — this can help identify jump targets.
- Screenshot: Game Boy native resolution is 160x144. PIL's NEAREST interpolation preserves pixel art when scaling.
- MCP image content format: `{"type": "image", "data": "<base64>", "mimeType": "image/png"}`.
