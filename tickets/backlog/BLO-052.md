# BLO-052: Game state machine detector

## Objective

Implement `gb_detect_states(duration_frames?, ram_address?)` — a tool that identifies game states and scenes by analyzing execution patterns and RAM changes over time. This is a research ticket; the approach is exploratory and may require multiple iterations to produce useful results.

## Context

Game Boy games typically use a state machine pattern: a RAM variable holds the current "scene" (title screen, gameplay, menu, transition, etc.) and the main loop dispatches to different routines based on that value. Identifying this structure automatically would greatly accelerate decompilation by revealing the high-level program flow. This tool combines execution tracing (BLO-037) with control flow graph analysis (BLO-035) to detect state transitions.

## Acceptance Criteria

- [ ] `gb_detect_states` MCP tool registered and callable
- [ ] Accepts optional `duration_frames` parameter (default: reasonable value, e.g., 300 frames / ~5 seconds)
- [ ] Accepts optional `ram_address` parameter to monitor a specific RAM location as the state variable
- [ ] When no `ram_address` given, heuristically identifies candidate state variables by detecting RAM addresses whose values correlate with PC hotspot changes
- [ ] Returns detected states with: state ID/value, representative PC ranges, frame ranges observed
- [ ] Detects transitions between states with frame numbers
- [ ] Domain module `domain/state_machine.py` contains all analysis logic
- [ ] Unit tests for domain module written before implementation (TDD)
- [ ] Tool handler validates inputs and delegates to domain

## Technical Approach

1. **Domain module (`domain/state_machine.py`)**:
   - `detect_state_variable(trace_data, ram_snapshots) -> list[CandidateStateVar]` — heuristic to find RAM addresses that act as state selectors
   - `detect_states(trace_data, ram_snapshots, state_address) -> StateMachineResult` — given a known state variable, segment the trace into states
   - Heuristics for state variable detection:
     - Low-cardinality values (small number of distinct values over the trace)
     - Value changes correlate with shifts in PC distribution (different code paths active)
     - Located in HRAM (0xFF80-0xFFFE) or WRAM (0xC000-0xDFFF)
   - State segmentation: group consecutive frames with the same state variable value

2. **Tool handler (`tools/gb_detect_states.py`)**:
   - Run emulator for `duration_frames`, collecting PC trace and RAM snapshots per frame
   - Uses tracing infrastructure from BLO-037
   - Delegate analysis to domain functions
   - Format and return results

3. **Tests (`tests/domain/test_state_machine.py`)**:
   - Synthetic trace data with known state variable — verify detection
   - Multiple candidate variables — verify ranking
   - State transitions detected at correct frame boundaries
   - Edge case: single state (no transitions)

## Dependencies

- BLO-037: Execution tracing infrastructure (provides PC trace data)
- BLO-035: Control flow graph analysis (provides structural context)

## Notes

- This is a **research** ticket — the heuristics may need tuning and the first implementation may not work well for all games.
- PyBoy's per-frame RAM access may be a performance bottleneck; consider sampling rather than capturing every frame.
- Future iterations could incorporate VRAM tile analysis to improve state detection (visual changes correlate with state changes).
- Consider limiting the candidate search space to known Game Boy state variable patterns (e.g., single-byte values in WRAM that are read in the main loop).
