# BLO-070: README overhaul with demo

## Objective

Rewrite `README.md` to be attractive and informative for new users discovering the project. The updated README should clearly communicate what blobert-mcp does, how to install and use it, and include a demo recording showing a ROM analysis session. Add standard project badges for CI status, Python version, and license.

## Context

The current README is basic and does not do justice to the 25 implemented MCP tools. As the project approaches v1.0, the README is the primary entry point for potential users and contributors. A polished README with a demo recording significantly improves first impressions and reduces onboarding friction.

## Acceptance Criteria

- [ ] README.md contains a clear, compelling project description (what it is, who it's for, why it exists)
- [ ] Feature highlights section lists key capabilities with brief descriptions
- [ ] Installation section covers `uv`-based setup with copy-pasteable commands
- [ ] Quick start section shows a minimal workflow from install to first ROM analysis
- [ ] Architecture overview section with a diagram or concise description of the system layers
- [ ] Badges added: CI status, Python version (3.10+), license
- [ ] Demo: terminal recording (asciinema or similar) embedded or linked, showing a ROM analysis session
- [ ] Contributing section with link to contribution guidelines or ticket system explanation
- [ ] README renders correctly on GitHub (all links, images, badges work)

## Technical Approach

- Restructure README.md with standard open-source project sections: description, badges, features, installation, quick start, architecture, contributing, license.
- Use shields.io for badges (CI status from GitHub Actions, Python version, license).
- Record a terminal demo using asciinema showing: loading a ROM, inspecting the header, disassembling a few instructions, adding an annotation. Host the recording on asciinema.org or embed as an SVG/GIF.
- Keep the architecture overview concise — link to `docs/architecture.md` for full details.
- Feature highlights should be organized by category (ROM analysis, emulation, knowledge base, etc.).
- Ensure all internal links (to docs/, etc.) are relative and work on GitHub.
