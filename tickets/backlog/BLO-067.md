# BLO-067: Docker image

## Objective

Provide a Docker image for easy deployment of blobert-mcp. The image should use a multi-stage build for minimal size, expose the stdio MCP transport, support optional volume mounts for ROM files and KB databases, and be published to GitHub Container Registry (ghcr.io) on release. Both amd64 and arm64 architectures must be supported.

## Context

A Docker image lowers the barrier to entry for users who want to run blobert-mcp without managing a Python environment. It also enables reproducible deployments and integrates well with container orchestration workflows. The MCP server uses stdio transport, so the container should be run interactively (`docker run -i`).

## Acceptance Criteria

- [ ] Dockerfile uses multi-stage build: build stage with uv for dependency installation, slim runtime stage
- [ ] `docker run -i blobert-mcp` starts the MCP server on stdio transport
- [ ] Optional volume mount points are documented for ROM files and KB databases
- [ ] GitHub Actions workflow builds and pushes to ghcr.io on release
- [ ] Multi-platform build supports both amd64 and arm64
- [ ] Image size is reasonable (under 200MB)

## Technical Approach

1. Create `Dockerfile` at project root:
   - **Build stage**: Use `python:3.10-slim` base, install `uv`, copy project files, run `uv sync --no-dev` to install dependencies.
   - **Runtime stage**: Use `python:3.10-slim` base, copy installed virtualenv from build stage. Set entrypoint to `blobert-mcp`.
   - Define volume mount points (e.g., `/data/roms`, `/data/kb`) for user convenience.
2. Create `.dockerignore` to exclude unnecessary files (`.git/`, `tests/`, `tickets/`, `docs/`, etc.).
3. Create or extend GitHub Actions workflow (`.github/workflows/docker.yml`):
   - Trigger on release publish events.
   - Use `docker/build-push-action` with QEMU for multi-platform builds (amd64 + arm64).
   - Push to `ghcr.io/<owner>/blobert-mcp`.
   - Tag with release version and `latest`.
4. Test locally: `docker build -t blobert-mcp . && echo '{}' | docker run -i blobert-mcp`.

## Notes

- The stdio transport means the container must be run with `-i` (interactive) to keep stdin open.
- Consider adding a `.dockerignore` to keep the build context small.
- GHCR authentication uses the built-in `GITHUB_TOKEN` — no additional secrets needed.
- Multi-platform builds via QEMU can be slow; consider using native runners if build time becomes an issue.
