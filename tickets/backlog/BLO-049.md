# BLO-049: C pseudocode decompiler

## Objective

Implement a `decompile_function(address)` MCP tool that transforms disassembled SM83 machine code into readable C-like pseudocode. The tool takes a function's entry address, reconstructs control flow using the CFG, resolves function/variable names from the knowledge base, and emits structured C-like output with comments. This is a research ticket — the approach may require multiple iterations to produce useful output.

## Context

Disassembly is a necessary first step, but raw SM83 instructions are difficult for humans to reason about at scale. A decompiler that lifts assembly into C-like pseudocode dramatically accelerates understanding of ROM logic. This builds on the control flow graph (BLO-035), cross-reference analysis (BLO-036), and the existing disassembly engine (BLO-005) to produce high-level representations of Game Boy functions. The knowledge base provides human-readable names for addresses and variables.

## Acceptance Criteria

- [ ] Domain module `domain/decompiler.py` exists with decompilation logic
- [ ] Control flow reconstruction converts CFG basic blocks into if/else, while, and for constructs
- [ ] Function calls are resolved to KB labels when available (e.g., `call $1234` becomes `PlaySound()`)
- [ ] Memory accesses use KB variable names when available (e.g., `[$C000]` becomes `player_x`)
- [ ] Register usage is tracked and simplified into local variable assignments where possible
- [ ] Output is a C-like pseudocode string with inline comments for unclear constructs
- [ ] MCP tool `decompile_function` is registered and callable with an `address` parameter
- [ ] Tool returns the pseudocode string or a structured error if the function cannot be decompiled
- [ ] Unit tests cover basic patterns: linear code, if/else, simple loops, function calls
- [ ] Edge cases are handled gracefully: indirect jumps, unresolved calls, overlapping functions

## Technical Approach

1. **Domain layer** (`src/blobert_mcp/domain/decompiler.py`):
   - **Lift instructions to IR**: Convert SM83 instructions into a simplified intermediate representation (assignments, conditionals, calls, returns).
   - **Recover control flow**: Use the CFG (from BLO-035) to identify structured patterns:
     - Single-entry/single-exit regions become sequential statements.
     - Conditional branches with merge points become if/else.
     - Back-edges in the CFG become while loops.
     - Counted loops (decrement + conditional jump) become for loops.
   - **Resolve names**: Query the KB for labels at call targets, memory addresses, and jump targets. Substitute symbolic names into the pseudocode.
   - **Simplify expressions**: Fold constant arithmetic, propagate register values where safe, eliminate redundant assignments.
   - **Emit C-like output**: Format the recovered structure as indented C-like pseudocode with comments marking uncertain decompilation.

2. **Tool layer** (`src/blobert_mcp/tools/decompile_function.py`):
   - Thin handler: validate address, invoke disassembler to get instructions, build CFG, call decompiler, format output.

3. **Testing** (TDD):
   - `tests/domain/test_decompiler.py` — synthetic instruction sequences covering linear code, branches, loops, and function calls.
   - `tests/tools/test_decompile_function.py` — tool integration tests.

4. **Iterative approach**: Given the research nature, start with linear-only decompilation, then add if/else, then loops. Each phase should be independently useful.

## Notes

- This is explicitly a research ticket. The SM83 instruction set has quirks (flag-dependent jumps, RST calls, HALT/STOP) that may not map cleanly to C constructs.
- Indirect jumps (`jp hl`) cannot be statically resolved — the decompiler should emit a comment noting the indirect jump rather than crashing.
- Consider an incremental output strategy: even partial decompilation (some blocks as pseudocode, others as raw assembly) is more useful than failing entirely.
- The quality of output depends heavily on KB annotations — the more labels available, the more readable the pseudocode.
- Future enhancement: integrate with the auto-annotation engine (BLO-048) to pre-populate names before decompilation.
