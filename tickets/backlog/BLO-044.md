# BLO-044: CGB-specific state tools

## Objective

Implement the `gb_get_cgb_state()` MCP tool to read Game Boy Color-specific hardware registers, providing visibility into CGB double speed mode, VRAM/WRAM bank selection, and HDMA transfer status. Done when the tool returns a structured JSON response with all CGB-specific register values and the implementation has full test coverage.

## Context

Game Boy Color ROMs use hardware features not present on the original DMG: double speed mode (toggled via KEY1), switchable VRAM and WRAM banks, and HDMA block/general-purpose DMA transfers. Without visibility into these registers, the LLM cannot determine which VRAM/WRAM bank is active or whether the CPU is running at double speed, making CGB ROM analysis unreliable. See `docs/mcp-spec.md` section 7 for the `gb_get_cgb_state` P2 tool specification.

## Acceptance Criteria

- [ ] `gb_get_cgb_state()` tool is registered as an MCP tool and callable with no parameters
- [ ] Reads KEY1 register (0xFF4D) and reports double speed mode status (bit 7) and speed switch armed flag (bit 0)
- [ ] Reads VBK register (0xFF4F) and reports current VRAM bank (0 or 1)
- [ ] Reads SVBK register (0xFF70) and reports current WRAM bank (1-7, with 0 mapped to 1)
- [ ] Reads HDMA registers (0xFF51-0xFF55) and reports HDMA source, destination, length, mode (general/HBlank), and active status
- [ ] Reports CGB flag from ROM header (0x0143) indicating CGB compatibility mode
- [ ] Returns `{"error": "NO_ROM_LOADED", ...}` when no ROM is loaded
- [ ] Domain logic is TDD -- tests written before implementation
- [ ] Tool handler is thin: validate, read registers, delegate to domain, format output

## Technical Approach

1. **Domain module** -- Add CGB state parsing functions in `domain/` (likely a new `cgb.py` or extend existing hardware module):
   - Parse KEY1 byte into `{double_speed: bool, switch_armed: bool}`
   - Parse VBK byte into VRAM bank number (bit 0)
   - Parse SVBK byte into WRAM bank number (bits 0-2, 0 maps to 1)
   - Parse HDMA1-5 into `{source: u16, destination: u16, length: int, mode: "general"|"hblank", active: bool, remaining_blocks: int}`
   - Parse CGB flag byte from header
2. **I/O register reads** -- Use `pyboy.memory[address]` for each register:
   - 0xFF4D (KEY1), 0xFF4F (VBK), 0xFF70 (SVBK)
   - 0xFF51 (HDMA1), 0xFF52 (HDMA2), 0xFF53 (HDMA3), 0xFF54 (HDMA4), 0xFF55 (HDMA5)
   - 0x0143 (CGB flag from ROM header)
3. **Tool handler** -- Register `gb_get_cgb_state` tool, read raw register bytes, delegate parsing to domain, return structured JSON
4. **Tests** -- TDD: write domain tests with synthetic register byte values first, then tool-level tests

## Notes

- HDMA registers behave differently when read vs. written. HDMA5 bit 7 indicates whether a transfer is active (0 = active). The remaining bits indicate blocks remaining.
- On DMG ROMs running in CGB mode, these registers may return unexpected values. Consider adding a warning if the ROM header CGB flag indicates DMG-only.
- VBK and SVBK only use their lower bits; upper bits should be masked.
