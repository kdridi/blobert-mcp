# BLO-038: KB struct and enum definitions

## Objective

Add struct and enum definition support to the knowledge base, enabling the LLM to define reusable data structures and enumerations and apply them to memory regions during ROM analysis. Done when `kb_define_struct`, `kb_apply_struct`, and `kb_define_enum` tools are implemented with SQLite persistence, domain validation, and full test coverage.

## Context

The knowledge base currently supports annotations, functions, and variables (BLO-005). Struct and enum definitions are the next layer of structured knowledge needed for effective decompilation. Game Boy ROMs heavily use packed data structures (e.g., sprite attribute tables, map headers, item records) and enumerated values (e.g., game states, item types, direction codes). Without struct/enum support, the LLM must re-derive field layouts every time it encounters the same data pattern. See `docs/mcp-spec.md` section 8 for the P1 tool specifications: `kb_define_struct`, `kb_apply_struct`, `kb_define_enum`.

## Acceptance Criteria

- [ ] New SQLite tables created: `structs`, `struct_fields`, `enums`, `enum_values`
- [ ] `kb_define_struct(name, fields)` tool defines a data structure with typed fields (each field has name, offset, type, size, comment); returns struct_id
- [ ] `kb_apply_struct(struct_name, address, count?)` tool applies a struct definition to a memory region; returns address-to-field mapping with byte values
- [ ] `kb_define_enum(name, values)` tool defines a named enumeration (value-to-name mapping); returns enum_id
- [ ] Domain validation in `domain/kb.py` (or new module) for struct fields (no overlapping offsets, valid types, positive sizes) and enum values (unique names, unique numeric values)
- [ ] Duplicate struct/enum names are rejected with clear error messages
- [ ] All domain logic is TDD — tests written before implementation
- [ ] Tool handlers in `tools/` are thin: validate input, delegate to domain, format output
- [ ] Existing KB functionality (annotations, functions, variables) is not broken

## Technical Approach

1. **Database schema** — Extend `kb/database.py` to add four new tables:
   - `structs`: id, name, total_size, comment, created_at
   - `struct_fields`: id, struct_id (FK), name, offset, type, size, comment
   - `enums`: id, name, comment, created_at
   - `enum_values`: id, enum_id (FK), name, value
2. **Domain validation** — Add validation functions in `domain/kb.py`:
   - Struct fields: no overlapping byte ranges, offset >= 0, size > 0, valid type strings
   - Enum values: unique names within enum, unique numeric values within enum
3. **KnowledgeBase class** — Add methods: `define_struct()`, `apply_struct()`, `define_enum()`
4. **Tool handlers** — Create tool handlers for the three new MCP tools following existing patterns
5. **Tests** — Write domain tests first (TDD), then integration tests for tool handlers

## Notes

- The `kb_apply_struct` tool requires a loaded ROM to read memory at the target address. It should return an error if no ROM is loaded.
- Consider whether enum types can be referenced in struct field type definitions (e.g., a field of type "enum:Direction"). This may be deferred to a follow-up ticket.
- The `count` parameter on `kb_apply_struct` enables array-of-struct interpretation (e.g., OAM table = 40 sprite entries).
