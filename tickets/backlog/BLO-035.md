# BLO-035: Control flow graph builder

## Objective

Implement the `get_control_flow_graph` P1 tool that builds a control flow graph (CFG) from a function entry point. The CFG decomposes a function into basic blocks connected by edges, enabling an LLM to understand conditional logic, loops, and branching structure — critical for decompilation.

## Context

The spec (docs/mcp-spec.md §1) defines `get_control_flow_graph` as a P1 tool: "Build the control flow graph of a function." With the disassembly engine complete (BLO-016) including `disassemble_range`, `disassemble_function`, and the full SM83 opcode table, all the building blocks are in place to analyze instruction flow and partition code into basic blocks.

## Acceptance Criteria

- [ ] `domain/cfg.py` module exists with pure functions for CFG construction
- [ ] `get_control_flow_graph(entry_point, max_depth?)` MCP tool builds a CFG from a function entry point
- [ ] Output includes list of basic blocks, each with: `start_addr`, `end_addr`, `instructions`
- [ ] Output includes list of edges, each with: `from_block`, `to_block`, `type` (one of `fall_through`, `jump`, `conditional`)
- [ ] Handles unconditional jumps (`JP nn`, `JP HL`) — terminate block, create `jump` edge to target
- [ ] Handles conditional branches (`JR NZ`, `JR Z`, `JR NC`, `JR C`, `JP NZ`, `JP Z`, `JP NC`, `JP C`) — terminate block, create `conditional` edge to target and `fall_through` edge to next instruction
- [ ] Handles `CALL` instructions — create edge to subroutine (marked as call edge), continue block or split depending on design
- [ ] Handles `RET` and `RETI` — terminate block as terminal (no outgoing edges)
- [ ] `max_depth` parameter limits analysis scope to prevent runaway exploration
- [ ] Domain module has no project imports (domain isolation rule)
- [ ] MCP tool handler is thin: validate input, delegate to domain, format output
- [ ] Tests written before implementation (TDD) — this is a complex algorithm requiring thorough testing
- [ ] Standard error format on failure (`{"error": "CODE", "message": "..."}`)

## Technical Approach

### Domain layer (`domain/cfg.py`)

1. **Basic block detection algorithm:**
   - Start at `entry_point`, disassemble instructions sequentially
   - A basic block ends when: (a) an unconditional jump/ret is reached, (b) a conditional branch is reached, (c) the current address is a known branch target from another block
   - Split blocks at branch targets discovered during analysis

2. **Edge construction:**
   - `fall_through`: next sequential instruction after a conditional branch
   - `jump`: target of an unconditional `JP` instruction
   - `conditional`: target of a conditional `JR`/`JP` instruction
   - `call`: target of a `CALL` instruction (optional — may be tracked separately)
   - Terminal blocks: `RET`, `RETI`, `JP HL` (indirect, target unknown)

3. **Data structures:**
   ```
   BasicBlock(start_addr, end_addr, instructions: list)
   Edge(from_block: int, to_block: int, type: str)
   CFG(blocks: list[BasicBlock], edges: list[Edge], entry_point: int)
   ```

4. **Algorithm sketch (worklist-based):**
   - Initialize worklist with `entry_point`
   - While worklist is not empty:
     - Pop address, disassemble forward until block-ending instruction
     - Record the block
     - Add branch targets to worklist (if not already processed)
     - Add edges based on instruction type
   - Post-process: split blocks at addresses that are branch targets of other blocks

### Tool layer (`tools/`)

- `get_control_flow_graph` tool: validate entry_point, optionally read bank, call domain CFG builder, serialize blocks and edges to JSON response

### Tests (`tests/domain/test_cfg.py`)

- Linear block (no branches) — single block, no edges
- Unconditional jump — two blocks, one `jump` edge
- Conditional branch — block splits into `conditional` + `fall_through` edges
- Loop detection — back-edge to earlier block
- `RET` as terminal — no outgoing edges
- `CALL` handling — edge to subroutine
- Multiple branch targets into same block — block gets split correctly
- `max_depth` limiting — stops exploration beyond limit
- `JP HL` (indirect jump) — terminal block (target unknown at static analysis time)

## Notes

- `JP HL` is an indirect jump whose target depends on runtime state. At static analysis time, the block should be marked as terminal with a note that the target is dynamic.
- `CALL` edges are conceptually different from intra-function branches. The design should distinguish them (possibly as a separate edge type) so the LLM can differentiate function calls from control flow within a single function.
- The `max_depth` parameter should limit the number of blocks explored, not the depth of the call graph. This prevents runaway analysis on large or recursive functions.
- This is a complex algorithm — TDD is mandatory per project conventions.
