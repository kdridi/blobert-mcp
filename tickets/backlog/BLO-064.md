# BLO-064: RAG — signature and engine detection

## Objective

Implement two MCP tools for identifying known patterns in Game Boy ROMs: `rag_find_signature` to match functions against known library signatures, and `rag_detect_engine` to identify the development toolchain used to build a ROM. Done when: known library functions (memcpy, strlen, div routines, etc.) can be identified by their assembly patterns, and the toolchain (GBDK, RGBDS, custom) can be detected by analyzing ROM-wide patterns.

## Context

Part of the Phase 7 RAG system described in `docs/mcp-spec.md`. While BLO-063 provides general similarity search, this ticket focuses on high-confidence identification of specific known patterns. Signature matching helps identify standard library functions that appear across many Game Boy ROMs. Engine detection helps set expectations about code structure, calling conventions, and available library functions for the entire ROM.

## Acceptance Criteria

- [ ] `rag_find_signature(function_address)` MCP tool registered and functional
- [ ] Signature tool reads bytes at the given address, canonicalizes, and matches against a curated signature database
- [ ] Returns matched signature with confidence score, function name, description, and source
- [ ] Returns empty/no-match result gracefully when no signature matches
- [ ] `rag_detect_engine(rom_path?)` MCP tool registered and functional
- [ ] Engine detection analyzes multiple patterns across the ROM: compiler-specific prologues, library code patterns, header conventions
- [ ] Returns detected toolchain (GBDK, RGBDS, custom, unknown) with confidence and evidence list
- [ ] Curated signature database includes common Game Boy library functions (memcpy, memset, strlen, div/mod routines, OAM DMA, VBlank wait)
- [ ] Engine fingerprint database includes patterns for GBDK and RGBDS toolchains
- [ ] Domain logic in `domain/` with no project imports; tool handlers are thin wrappers in `tools/`
- [ ] Tests written (TDD) for signature matching, engine detection, and edge cases

## Technical Approach

1. **Signature database**: A curated collection of known Game Boy library function patterns, stored as canonicalized assembly sequences with metadata (function name, description, source project). Can be stored in the same ChromaDB instance (separate collection) or as a static data file.
2. **`rag_find_signature` tool**:
   - Read bytes at the given function address from the loaded ROM
   - Disassemble to SM83 assembly
   - Canonicalize using BLO-065's pipeline
   - Search the signature database for high-similarity matches
   - Return matches above a confidence threshold
3. **Engine fingerprint database**: A set of heuristic rules and patterns that identify toolchains:
   - **GBDK**: C runtime prologue patterns, specific library function sequences, SDCC-generated code patterns
   - **RGBDS**: Macro expansion patterns, specific section layouts, common RGBDS library patterns
   - **Custom**: Absence of known toolchain patterns
4. **`rag_detect_engine` tool**:
   - Scan ROM header for toolchain hints
   - Sample code at known entry points and common addresses
   - Match patterns against engine fingerprint database
   - Aggregate evidence and report the most likely toolchain
5. **Key files**:
   - `src/blobert_mcp/domain/rag/signatures.py` — signature matching logic
   - `src/blobert_mcp/domain/rag/engine.py` — engine detection logic
   - `src/blobert_mcp/tools/rag_find_signature.py` — MCP tool handler
   - `src/blobert_mcp/tools/rag_detect_engine.py` — MCP tool handler
   - `data/signatures/` — curated signature patterns
   - `data/fingerprints/` — engine fingerprint rules
   - `tests/domain/rag/test_signatures.py` — signature matching tests
   - `tests/domain/rag/test_engine.py` — engine detection tests

## Notes

- Signature matching requires high-confidence thresholds — a false positive (misidentifying a function) is worse than a false negative (not identifying it).
- The curated signature database will start small and grow over time. Initial set should cover the most common and unambiguous patterns.
- Engine detection is inherently heuristic. Report confidence levels and supporting evidence so the user can judge reliability.
- GBDK detection is likely easier than RGBDS detection because GBDK/SDCC produces more distinctive code patterns.
- Consider whether signature patterns should live in the vector DB (dynamic, searchable) or as static data files (versioned, curated). A hybrid approach may be best.
