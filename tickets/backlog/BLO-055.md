# BLO-055: MCP resources and prompts

## Objective

Expose ROM data as MCP Resources and provide pre-configured MCP Prompts that guide LLMs through common decompilation workflows. Definition of done: resources are accessible via MCP resource URIs, prompts are listed and usable through the MCP prompt interface, and both integrate with the existing tool ecosystem.

## Context

MCP defines three primitives: Tools, Resources, and Prompts. The server currently only exposes Tools. Adding Resources gives LLMs structured, read-only access to ROM metadata and KB state without requiring tool calls. Adding Prompts provides reusable workflow templates that chain multiple tools into coherent analysis sessions. Together, these enhance the LLM's ability to reason about Game Boy ROMs with less back-and-forth.

## Acceptance Criteria

- [ ] Resource `rom://header` returns parsed ROM header (title, cartridge type, ROM/RAM size, checksums)
- [ ] Resource `rom://memory_map` returns the memory map layout of the loaded ROM
- [ ] Resource `rom://vector_table` returns interrupt and reset vector addresses with labels
- [ ] Resource `kb://annotations` returns all KB annotations for the loaded ROM
- [ ] Resource `kb://functions` returns all KB function entries for the loaded ROM
- [ ] Prompt `analyze_function` provides a disassemble + annotate workflow template
- [ ] Prompt `identify_routine` provides a run + observe + label workflow template
- [ ] Prompt `decompile_overview` provides a full ROM survey workflow template
- [ ] Resources and prompts use FastMCP `@resource` and `@prompt` decorators
- [ ] Resources return appropriate MIME types (application/json for structured data)
- [ ] Domain logic for resource data extraction is covered by tests (TDD)

## Technical Approach

1. **Domain layer** (`src/blobert_mcp/domain/`):
   - Resource data extraction functions: `get_rom_header()`, `get_memory_map()`, `get_vector_table()`, `get_kb_annotations()`, `get_kb_functions()`.
   - These are pure functions that take ROM bytes / KB state and return structured dicts.

2. **Resource registration** (`src/blobert_mcp/tools/` or a new `src/blobert_mcp/resources/`):
   - Use FastMCP `@mcp.resource("rom://header")` decorator pattern.
   - Each resource handler delegates to domain functions, returns JSON.

3. **Prompt registration** (`src/blobert_mcp/prompts/` or alongside resources):
   - Use FastMCP `@mcp.prompt("analyze_function")` decorator pattern.
   - Each prompt returns a structured message sequence that references available tools.
   - `analyze_function`: takes an address, chains disassembly -> annotation -> summary.
   - `identify_routine`: takes an address, chains execution -> observation -> labeling.
   - `decompile_overview`: no args, chains header read -> memory map -> function scan -> summary.

4. **Server wiring** (`src/blobert_mcp/server.py`):
   - Import and register resources and prompts with the FastMCP server instance.

## Dependencies

- BLO-013: Execution engine needed for prompts that reference execution tools.
- BLO-005: KB system needed for `kb://` resources and annotation-related prompts.

## Notes

- FastMCP resource and prompt APIs should be verified against the current FastMCP version before implementation.
- Resources are read-only by MCP spec — they should never mutate state.
- Prompts are templates, not executors — the LLM decides whether to follow the suggested workflow.
- Consider whether `kb://` resources should support parameterized URIs (e.g., `kb://annotations?bank=0x01`) in a future ticket.
