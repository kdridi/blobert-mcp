# BLO-029: Call stack reconstruction

## Objective

Implement `gb_get_call_stack` MCP tool that performs heuristic SP-based call stack reconstruction for the SM83 CPU. After this ticket, the LLM can inspect the current execution context when stopped at a breakpoint or after stepping, seeing the chain of return addresses on the stack.

## Context

The SM83 CPU has no frame pointer register — there is only SP. When a `CALL` instruction executes, it pushes the return address (PC + 3) onto the stack and jumps to the target. `RET` pops that address and jumps back. However, `PUSH` instructions also place data on the stack, so not every 16-bit value on the stack is a return address.

The spec (section 2, `gb_get_call_stack`) defines this as P1 and explicitly notes the heuristic nature: "read stack memory from SP upward, cross-reference return addresses with known function boundaries in the KB. Heuristic — may include false positives on non-CALL stack entries."

The reconstruction algorithm must validate candidate return addresses by checking that the bytes preceding the candidate address in ROM correspond to a `CALL` instruction (opcode `0xCD`, `0xC4`, `0xCC`, `0xD4`, `0xDC`). This filters out most `PUSH`ed values.

## Acceptance Criteria

- [ ] Domain module `domain/callstack.py` exists with pure heuristic call stack reconstruction logic
- [ ] The heuristic reads 16-bit values from SP upward and validates each as a potential return address by checking that the preceding bytes in memory are a CALL instruction
- [ ] `gb_get_call_stack(max_depth)` tool is registered and returns a list of return addresses with optional KB labels
- [ ] The tool respects the `max_depth` parameter (default 16)
- [ ] False positives from `PUSH` instructions are filtered out by the CALL-instruction validation
- [ ] Unit tests for `domain/callstack.py` cover: empty stack, single CALL, nested CALLs, PUSH values correctly filtered, max_depth limit
- [ ] Tests use synthetic byte data (no real ROMs)
- [ ] Tool follows the existing error format: `{"error": "CODE", "message": "..."}`

## Technical Approach

1. **Domain module `domain/callstack.py`** (TDD — tests first):
   - Pure function: `reconstruct_call_stack(stack_bytes, sp, memory_reader, max_depth) -> list[CallFrame]`
   - `stack_bytes`: raw bytes from SP to top of stack (0xFFFE)
   - `memory_reader`: callable to read ROM bytes at an address (for CALL validation)
   - For each 16-bit value on the stack (little-endian, starting at SP):
     - Treat it as a candidate return address
     - Read the 3 bytes preceding that address in ROM
     - Check if byte[0] is a CALL opcode (`0xCD` for unconditional, `0xC4`/`0xCC`/`0xD4`/`0xDC` for conditional)
     - If valid, add to call stack with the CALL target and return address
   - Stop when `max_depth` is reached or stack is exhausted

2. **Tool handler `tools/callstack.py`**:
   - Read SP from registers
   - Read stack memory from SP to 0xFFFE using `gb_read_memory`
   - Read ROM bytes as needed for CALL validation
   - Delegate to domain function
   - Enrich results with KB labels if available

3. **Tests**: Construct synthetic stack contents and ROM bytes, verify correct reconstruction.

## Notes

- SM83 CALL opcodes: `0xCD` (CALL nn), `0xC4` (CALL NZ,nn), `0xCC` (CALL Z,nn), `0xD4` (CALL NC,nn), `0xDC` (CALL C,nn). All are 3 bytes long.
- RST instructions (`0xC7`, `0xCF`, `0xD7`, `0xDF`, `0xE7`, `0xEF`, `0xF7`, `0xFF`) also push a return address but are only 1 byte long. The heuristic should also check for RST at `return_addr - 1`.
- Stack grows downward on SM83: SP decreases on PUSH/CALL. Scanning upward from SP means increasing addresses.
- This is inherently heuristic — the tool output should indicate confidence or mark entries as "uncertain" when validation is ambiguous.
