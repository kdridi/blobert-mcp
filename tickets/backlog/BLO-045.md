# BLO-045: Memory access logging

## Objective

Implement the `gb_get_memory_access_log(address_range, duration_frames)` MCP tool to detect memory writes to a specified address range over a given number of frames using polling-based change detection. Done when the tool returns a list of changes with frame number, address, old value, and new value, and the implementation has full test coverage.

## Context

Understanding which code modifies a memory location is fundamental to ROM decompilation. PyBoy does not provide memory watchpoint APIs, so true read/write interception is not possible. Instead, this tool uses a polling approach: snapshot the target memory range each frame, compare against the previous snapshot, and record any changes. This only detects writes (by delta), not reads. See `docs/mcp-spec.md` section 3 (`gb_get_memory_access_log`, P2) for the full specification, including the feasibility warning about polling-based limitations.

## Acceptance Criteria

- [ ] `gb_get_memory_access_log(address, length, duration_frames, type?)` tool is registered as an MCP tool
- [ ] Accepts `address` (start), `length` (range size), `duration_frames` (how many frames to monitor)
- [ ] `type` parameter accepted but only `"write"` and `"both"` produce results (reads are undetectable via polling); documents this limitation in the response
- [ ] Snapshots the memory range before each frame tick, compares after tick, records deltas
- [ ] Returns list of `{frame: int, address: int, old_value: int, new_value: int}` entries
- [ ] Respects output limits (max 256 entries by default, with `truncated` and `total` fields if exceeded)
- [ ] Returns `{"error": "NO_ROM_LOADED", ...}` when no ROM is loaded
- [ ] Validates input: address range within valid memory, length <= 4096, duration_frames > 0 and <= 3600
- [ ] Domain logic is TDD -- tests written before implementation
- [ ] Tool handler is thin: validate input, delegate to domain/session, format output

## Technical Approach

1. **Domain module** -- Add memory diffing logic (likely in `domain/memory.py` or a new `domain/memory_log.py`):
   - `diff_memory_snapshots(before: bytes, after: bytes, base_address: int) -> list[MemoryChange]`
   - `MemoryChange` dataclass: `frame`, `address`, `old_value`, `new_value`
2. **Session integration** -- Add a method to `EmulatorSession` that:
   - Takes initial snapshot of the address range
   - Loops for `duration_frames`, calling `pyboy.tick()` each iteration
   - After each tick, snapshots again and diffs against previous
   - Collects all `MemoryChange` entries, respecting the max limit
   - Returns the collected log
3. **Tool handler** -- Register `gb_get_memory_access_log` tool, validate parameters, delegate to session method, format output JSON
4. **Tests** -- TDD for domain diffing logic with synthetic byte arrays. Integration tests for the session polling loop using mock PyBoy.

## Notes

- Performance consideration: snapshotting and diffing every frame for large ranges (e.g., 4096 bytes) over many frames could be slow. Consider documenting performance characteristics or adding a `sample_interval` parameter in a follow-up.
- The spec notes a V1 alternative: use `gb_memory_diff` between save states. This tool is the more granular approach.
- Reads cannot be detected via polling. If `type="read"` is requested, return an informative error explaining the limitation.
- Multiple writes to the same address within a single frame will only show the net result (before-tick vs after-tick), not intermediate values.
