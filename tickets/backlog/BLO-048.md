# BLO-048: Auto-annotation engine

## Objective

Implement a `kb_auto_annotate(strategy?)` MCP tool that automatically labels known patterns in a loaded ROM using heuristic-based detection algorithms. When complete, the tool will scan the ROM for recognizable structures (interrupt vectors, string tables, jump tables, VBlank handlers) and create knowledge base annotations for each discovery.

## Context

With 25 MCP tools implemented including a full disassembly engine and SQLite-backed knowledge base, the next productivity gain is automating the tedious manual annotation of well-known ROM structures. Game Boy ROMs follow predictable conventions (RST vectors at fixed addresses, interrupt handlers, ASCII string tables, jump table patterns) that can be detected mechanically, freeing the human analyst to focus on game-specific logic.

## Acceptance Criteria

- [ ] Domain module `domain/auto_annotate.py` exists with detection algorithms for each strategy
- [ ] Strategy "vectors" labels RST handlers (RST $00-$38) and interrupt handlers (VBlank $0040, LCD STAT $0048, Timer $0050, Serial $0058, Joypad $0060)
- [ ] Strategy "strings" scans ROM for contiguous printable ASCII sequences and labels them as string tables
- [ ] Strategy "jump_tables" detects patterns of consecutive absolute addresses and labels them as jump tables
- [ ] Strategy "vblank" identifies the VBlank handler routine and labels it
- [ ] Strategy "all" runs every strategy in sequence
- [ ] Each detected pattern creates a KB annotation with appropriate type and descriptive comment
- [ ] MCP tool `kb_auto_annotate` is registered and callable with optional `strategy` parameter (defaults to "all")
- [ ] Tool returns summary of annotations created (count per strategy, addresses annotated)
- [ ] Unit tests cover each strategy with synthetic ROM byte sequences
- [ ] Existing KB annotations are not duplicated on repeated runs

## Technical Approach

1. **Domain layer** (`src/blobert_mcp/domain/auto_annotate.py`):
   - Define a `Strategy` enum: `VECTORS`, `STRINGS`, `JUMP_TABLES`, `VBLANK`, `ALL`.
   - Implement one detection function per strategy, each taking ROM bytes and returning a list of `Annotation` dataclass instances (address, label, type, comment).
   - `detect_vectors(rom_bytes)` — check fixed addresses for non-NOP instructions.
   - `detect_strings(rom_bytes, min_length=4)` — scan for contiguous printable ASCII runs.
   - `detect_jump_tables(rom_bytes)` — look for sequences of 2-byte absolute addresses pointing into ROM.
   - `detect_vblank(rom_bytes)` — analyze the $0040 vector target to identify the full handler routine.
   - `run_strategies(rom_bytes, strategy)` — dispatcher that runs selected or all strategies.

2. **Tool layer** (`src/blobert_mcp/tools/kb_auto_annotate.py`):
   - Thin handler: validate input, load ROM bytes, call `run_strategies()`, write annotations to KB via existing KB infrastructure, format summary output.

3. **Testing** (TDD):
   - `tests/domain/test_auto_annotate.py` — synthetic ROM bytes exercising each detection algorithm.
   - `tests/tools/test_kb_auto_annotate.py` — tool integration tests.

## Notes

- String detection should have a configurable minimum length to avoid false positives (short sequences of bytes that happen to be printable ASCII).
- Jump table detection is inherently heuristic — expect some false positives. Consider a confidence threshold.
- Idempotency is important: running the tool twice should not create duplicate annotations.
