# BLO-065: RAG — index annotated pairs

## Objective

Build the foundation of the RAG system by implementing the `rag_index_pair` tool and the underlying vector database infrastructure. This tool allows adding raw-to-annotated assembly pairs to a reference database, enabling future similarity search and signature matching. Done when: pairs can be indexed, persisted in a vector DB, and retrieved by embedding similarity; a batch indexing script can populate the DB from pokered/pokecrystal `.sym` + `.asm` files.

## Context

The MCP spec (Phase 7) describes a RAG system for leveraging annotated Game Boy code from existing decompilation projects (pokered, pokecrystal, etc.). This ticket establishes the vector database, embedding pipeline, and indexing tool that BLO-063 (similar code search) and BLO-064 (signature/engine detection) depend on. The RAG database is separate from the main knowledge base — different database, different purpose.

## Acceptance Criteria

- [ ] Vector DB (ChromaDB or similar lightweight Python-native solution) is set up with a schema for storing raw/annotated assembly pairs
- [ ] Embedding pipeline implemented: instruction canonicalization (replace specific addresses/labels with placeholders) followed by text embedding
- [ ] `rag_index_pair(raw_asm, annotated_asm, source?, metadata?)` MCP tool registered and functional
- [ ] Tool accepts raw assembly, annotated assembly, optional source project name, and optional metadata dict
- [ ] Indexed pairs are persisted and retrievable by embedding similarity
- [ ] Batch indexing script can parse pokered/pokecrystal `.sym` + `.asm` files and populate the DB
- [ ] Domain logic is in `domain/` with no project imports; tool handler is a thin wrapper in `tools/`
- [ ] Tests written (TDD) for canonicalization, embedding, indexing, and retrieval
- [ ] Architecture documented in `docs/decisions.md` (vector DB choice, embedding strategy)

## Technical Approach

1. **Vector DB setup**: Use ChromaDB (pure Python, no external server). Create a dedicated collection for assembly pairs with metadata fields for source project, file path, and function name.
2. **Canonicalization**: Domain function to normalize SM83 assembly — replace concrete addresses (`$4A3F`) with placeholders (`$ADDR`), replace label names with generic tokens, normalize whitespace and comments. This ensures semantically similar code produces similar embeddings.
3. **Embedding pipeline**: Canonicalize the raw assembly, then generate a text embedding. Start with ChromaDB's default embedding function; evaluate alternatives if quality is insufficient.
4. **`rag_index_pair` tool**: Thin handler that validates inputs, calls domain canonicalization, stores the pair in ChromaDB with metadata.
5. **Batch script**: CLI script or management tool that walks a decompilation project directory, pairs `.asm` blocks with `.sym` annotations, and calls the indexing pipeline.
6. **Key files**:
   - `src/blobert_mcp/domain/rag/` — canonicalization, embedding, DB abstraction
   - `src/blobert_mcp/tools/rag_index_pair.py` — MCP tool handler
   - `scripts/rag_batch_index.py` — batch population script
   - `tests/domain/rag/` — unit tests

## Notes

- ChromaDB is the leading candidate for vector DB — pure Python, embeds in-process, supports persistence, includes a default embedding function. Evaluate at implementation time.
- The canonicalization strategy is critical for search quality. Over-canonicalization loses specificity; under-canonicalization misses similar patterns. Will need iteration.
- Batch indexing from pokered/pokecrystal requires understanding their file layout (`.asm` source files + `.sym` symbol files). These are open-source projects.
- This database is intentionally separate from any main knowledge base to keep concerns isolated.
