# BLO-036: Cross-reference analysis tool

## Objective

Implement `get_cross_references(address, direction?)` to find all references to or from a given address in the ROM. This enables the LLM to answer questions like "who calls this function?" and "what does this function reference?" by scanning the entire ROM, disassembling it, and building a cross-reference index. Done when the tool returns accurate cross-references with type classification and integrates with the knowledge base for label display.

## Context

Cross-reference analysis is a fundamental reverse engineering capability listed as P1 in `docs/mcp-spec.md` section 1 (`get_cross_references`). Without it, the LLM must manually trace call chains, which is impractical for large ROMs. The disassembly engine (BLO-016) and ROM reading infrastructure (BLO-005) are complete, providing the foundation for a full ROM scan.

From the spec:
- **Input:** `address`, `direction` ("to" | "from" | "both")
- **Output:** list of source addresses with type (CALL, JP, JR, LD), max 100 results
- **Usage:** "who calls this function?", "who reads this variable?"
- **Implementation:** requires full ROM scan

## Acceptance Criteria

- [ ] Domain module `domain/xrefs.py` exists with pure functions for ROM scanning and cross-reference index building
- [ ] Full ROM scan disassembles all banks and identifies CALL, JP, JR, and LD instructions with address operands
- [ ] Cross-reference index maps each target address to a list of referencing addresses with instruction type
- [ ] `direction="to"` returns all addresses that reference the given address (callers/jumpers to it)
- [ ] `direction="from"` returns all addresses that the given address references (callees/jump targets from it)
- [ ] `direction="both"` returns references in both directions
- [ ] Results are capped at 100 entries per the spec, with `truncated` and `total` fields when applicable
- [ ] Integration with the knowledge base: results include KB labels when available for referenced addresses
- [ ] Tool handler `get_cross_references` registered as an MCP tool with proper validation and error handling
- [ ] Unit tests for `domain/xrefs.py` using synthetic byte data (TDD)
- [ ] Tool-level tests for the MCP handler

## Technical Approach

1. **Domain module `domain/xrefs.py`:**
   - `scan_rom_for_xrefs(read_bytes_fn, bank_count, bank_size) -> XRefIndex` — scans all ROM banks, disassembles each instruction, and extracts address operands from CALL, JP, JR, and LD instructions.
   - `XRefIndex` dataclass holding two dicts: `refs_to[address] -> list[XRef]` and `refs_from[address] -> list[XRef]`.
   - `XRef` dataclass: `source_address`, `target_address`, `instruction_type` (CALL/JP/JR/LD), `source_bank`, `target_bank`.
   - `query_xrefs(index, address, direction) -> list[XRef]` — filters the index by direction.

2. **Index building strategy:**
   - Iterate over all ROM banks (bank 0 at 0x0000-0x3FFF, banks 1..N at 0x4000-0x7FFF).
   - Use the existing disassembly engine to decode each instruction.
   - For branch/call instructions, resolve the target address (absolute for CALL/JP, relative for JR).
   - For LD instructions with 16-bit immediate operands pointing into ROM/RAM ranges, record as data references.
   - Cache the index so repeated queries don't require re-scanning.

3. **Tool handler in `tools/`:**
   - Validate inputs (address range, direction enum).
   - Build or retrieve cached xref index.
   - Query the index and format results.
   - Enrich results with KB labels if the knowledge base is available.

4. **Testing (TDD):**
   - Write tests first for `domain/xrefs.py` using synthetic ROM bytes containing known CALL/JP/JR/LD patterns.
   - Test edge cases: cross-bank references, JR wraparound, LD with non-address immediates.

## Notes

- Full ROM scan may be slow for large ROMs (up to 8MB / 512 banks). Consider lazy or incremental indexing strategies.
- The xref index should be invalidated if a different ROM is loaded.
- JR instructions use signed 8-bit offsets relative to the instruction following the JR; handle the sign extension correctly.
- LD instructions referencing addresses in the 0x0000-0x7FFF range are ROM references; those in 0xC000-0xDFFF are RAM references. Both are useful for cross-referencing.
