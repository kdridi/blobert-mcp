# BLO-039: KB export multi-format

## Objective

Implement the `kb_export` tool to export the knowledge base in multiple formats useful for decompilation workflows. Done when the tool supports "sym" (RGBDS symbol file), "asm" (assembly source with labels), "c_header" (C header with defines/structs), and "json" (full KB dump) formats, with domain logic in a dedicated module and full test coverage.

## Context

Once the LLM has built up annotations, function definitions, and variable names in the knowledge base, that knowledge needs to be exportable to standard formats used by the Game Boy development and reverse-engineering community. RGBDS .sym files are the de facto standard for Game Boy symbol files, assembly output with labels enables reading decompiled code, C headers support GBDK-based projects, and JSON provides a complete machine-readable dump. See `docs/mcp-spec.md` section 8 for the `kb_export` specification.

## Acceptance Criteria

- [ ] `kb_export(format)` tool implemented, accepting format as "sym", "asm", "c_header", or "json"
- [ ] "sym" format: produces valid RGBDS symbol file output (`;` comments, `bank:address label` lines)
- [ ] "asm" format: produces assembly source with labels, function boundaries, and inline comments from annotations
- [ ] "c_header" format: produces C header with `#define` for constants, `typedef struct` for defined structs, `enum` for enums, and `extern` declarations for variables
- [ ] "json" format: produces complete KB dump (all annotations, functions, variables, and any structs/enums if BLO-038 is done)
- [ ] Invalid format parameter returns a clear error
- [ ] Domain module `domain/kb_export.py` contains all format generation logic (pure functions, no I/O)
- [ ] All domain logic is TDD — tests written before implementation
- [ ] Tool handler is thin: validates format, calls domain, returns formatted string
- [ ] Export works correctly with an empty knowledge base (produces valid but minimal output)

## Technical Approach

1. **Domain module** — Create `src/blobert_mcp/domain/kb_export.py` with pure functions:
   - `export_sym(annotations, functions, variables)` -> str
   - `export_asm(annotations, functions, variables)` -> str
   - `export_c_header(annotations, functions, variables, structs?, enums?)` -> str
   - `export_json(annotations, functions, variables, structs?, enums?)` -> str
   - Each function takes data as plain dataclasses/dicts — no database dependency
2. **KnowledgeBase method** — Add `export(format)` method that queries all relevant data and delegates to the domain module
3. **Tool handler** — Register `kb_export` MCP tool, validate the format parameter, call `KnowledgeBase.export()`, return the string content
4. **Format details**:
   - **sym**: `; blobert-mcp export\n` header, then `BB:XXXX label` per annotation/function sorted by address
   - **asm**: `SECTION` directives, labels at function starts, `; comment` lines from annotations, `.db`/`.dw` for data regions
   - **c_header**: standard `#ifndef` guard, `#define ADDR_name 0xXXXX` for addresses, struct/enum typedefs if available
   - **json**: `{"annotations": [...], "functions": [...], "variables": [...]}` with all fields

## Notes

- The c_header and asm formats can optionally include struct/enum definitions if BLO-038 is completed. The export should gracefully handle missing struct/enum tables (they may not exist yet).
- RGBDS sym format reference: lines are `BB:XXXX symbolname` where BB is the bank in hex and XXXX is the address in hex.
- Consider output size limits for large knowledge bases — the spec mentions output limits with truncation.
