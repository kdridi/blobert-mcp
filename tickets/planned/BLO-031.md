# BLO-031: Input sequence playback and recording

## Objective

Implement two MCP tools for deterministic input sequence playback and input recording during execution. `gb_play_input_sequence` accepts a list of timestamped button events and plays them frame-by-frame through the emulator. `gb_record_input_sequence` captures inputs during a specified number of frames. Together these enable TAS-style deterministic replay and input capture workflows for ROM analysis.

## Context

The emulator currently supports single button press/release via `gb_press_button` (P0, implemented). For effective ROM exploration, the LLM needs to replay complex input sequences deterministically (e.g., navigating menus, triggering specific game states). See `docs/mcp-spec.md` section 5 ("Inputs and input sequences") for the full specification. `gb_play_input_sequence` is P1 and `gb_record_input_sequence` is P2.

## Acceptance Criteria

- [ ] `gb_play_input_sequence(sequence, max_frames)` tool implemented and registered
- [ ] Sequence format: list of `{frame: int, button: str, action: "press"|"release"}`
- [ ] `max_frames` parameter enforced (default 3600 = ~60s) to prevent runaway execution
- [ ] Returns final state: registers, PC, and screenshot after sequence completes
- [ ] `gb_record_input_sequence(duration_frames)` tool implemented and registered
- [ ] Recording returns captured sequence in the same `{frame, button, action}` format
- [ ] Domain module for sequence validation (frame ordering, valid buttons, valid actions)
- [ ] Domain validation rejects invalid button names, invalid actions, out-of-order frames, and frames exceeding max_frames
- [ ] Tests written for domain validation logic (TDD)
- [ ] Tests written for tool handlers
- [ ] Error handling: returns proper error format for NO_ROM_LOADED, invalid sequence data

## Technical Approach

1. **Domain module** (`domain/input_sequence.py`):
   - Define `InputEvent` dataclass: `frame: int`, `button: str`, `action: str`
   - `validate_sequence(sequence, max_frames)` — validates ordering, button names, action values, frame bounds
   - `parse_sequence(raw_data)` — converts raw dicts to `InputEvent` list with validation
   - Valid buttons: `a`, `b`, `start`, `select`, `up`, `down`, `left`, `right`
   - Valid actions: `press`, `release`

2. **Tool: `gb_play_input_sequence`** (`tools/input_sequence.py`):
   - Accept `sequence` (list of event dicts) and `max_frames` (int, default 3600)
   - Delegate validation to domain module
   - Iterate frame-by-frame, applying button press/release events at their designated frames via PyBoy API
   - After completion, return registers, PC, and a screenshot

3. **Tool: `gb_record_input_sequence`** (`tools/input_sequence.py`):
   - Accept `duration_frames` (int)
   - Run emulator for the specified duration, capturing any button events
   - Return the recorded sequence in the standard format

4. **Tests** (TDD — write before implementation):
   - `tests/domain/test_input_sequence.py` — validation edge cases
   - `tests/tools/test_input_sequence.py` — tool handler behavior

## Notes

- The sequence format mirrors TAS movie files — each event specifies an absolute frame number and a button action.
- `gb_play_input_sequence` is P1 (important for analysis workflows). `gb_record_input_sequence` is P2 (nice-to-have for human-player mode) but included here since it shares the same domain module and is low marginal effort.
- Frame-by-frame iteration uses `pyboy.tick()` per frame, applying events at the correct frame count.
