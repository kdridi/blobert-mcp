# BLO-033: VRAM, sprite, and tilemap inspection tools

## Objective

Implement four visual/multimodal inspection tools (`gb_inspect_vram`, `gb_list_sprites`, `gb_get_tilemap`, `gb_read_palettes`) that expose the Game Boy's graphics state to the LLM. These tools let the LLM see tile data in VRAM, enumerate OAM sprites, read background/window tilemaps, and inspect palette registers. When complete, all four spec section 6 inspection tools will be available alongside the existing `gb_screenshot` tool.

## Context

The spec (section 6 — Visual and multimodal) defines these as P1/P2 tools that extend the LLM's ability to understand what the Game Boy is rendering and why. With `gb_screenshot` (BLO-022) already implemented using PIL and MCP Image output, these tools follow the same visual output pattern. PyBoy provides tilemap and sprite APIs (`pyboy.tilemap_background()`, `pyboy.tilemap_window()`, `pyboy.sprite()`) that should be leveraged for data extraction. A new domain module (`domain/graphics.py`) will encapsulate tile/sprite data structures and OAM parsing logic, keeping business logic out of tool handlers.

## Acceptance Criteria

- [ ] `gb_inspect_vram()` renders all 384 tiles as a composite image and returns JSON metadata (tile index, address) alongside an MCP Image
- [ ] `gb_list_sprites()` reads all 40 OAM entries and returns x, y, tile_id, attributes (flip, palette, priority), and visibility status
- [ ] `gb_get_tilemap(layer)` extracts the background or window tilemap (32x32 tile indices) and returns JSON data with an optional grid overlay image
- [ ] `gb_read_palettes()` reads DMG palettes (BGP, OBP0, OBP1) from I/O registers and CGB palettes (8 BG + 8 OBJ) via indexed I/O
- [ ] Domain module `domain/graphics.py` contains pure data structures and parsing logic for tiles, sprites, OAM, and palettes
- [ ] Domain module has no imports from the project (domain isolation rule)
- [ ] TDD: domain-layer tests written before implementation in `tests/domain/test_graphics.py`
- [ ] Integration tests for all four tool handlers in `tests/tools/`
- [ ] Visual outputs use PIL + MCP Image type (same pattern as `gb_screenshot`)

## Technical Approach

### Domain layer (`src/blobert_mcp/domain/graphics.py`)

Define pure data structures and parsing functions:

- `TileInfo(index, address, pixel_data)` — metadata for a single 8x8 tile
- `SpriteEntry(index, x, y, tile_id, flags, on_screen)` — parsed OAM entry with attribute flags (x_flip, y_flip, palette, priority)
- `TilemapData(layer, width, height, tile_indices)` — 32x32 grid of tile IDs
- `PaletteData` — DMG (BGP, OBP0, OBP1 as 4-shade arrays) and CGB (8 BG + 8 OBJ palettes as RGB arrays)
- `parse_oam_entry(bytes) -> SpriteEntry` — parse 4 OAM bytes into a sprite entry
- `parse_dmg_palette(register_value) -> list[int]` — decode a DMG palette register into 4 shade values
- `parse_cgb_palette(palette_bytes) -> list[tuple]` — decode CGB palette data into RGB tuples

### Tool handlers (`src/blobert_mcp/tools/`)

Each tool handler follows the thin-tool pattern: validate inputs, read bytes from PyBoy, delegate to domain functions, format output.

1. **`gb_inspect_vram`** — Use PyBoy's tile data access to read all 384 tiles (3 tile blocks x 128 tiles). Render as a 16x24 grid of 8x8 tiles using PIL. Return composite image as MCP Image + JSON metadata array.

2. **`gb_list_sprites`** — Use `pyboy.sprite(index)` for indices 0-39 to read OAM. Extract x, y, tile_identifier, attributes, on_screen. Return JSON list of all 40 entries.

3. **`gb_get_tilemap`** — Use `pyboy.tilemap_background()` or `pyboy.tilemap_window()` based on `layer` parameter. Read `tile_identifier(col, row)` for all 32x32 positions. Optionally render grid overlay on the tilemap image using PIL. Return JSON grid + optional MCP Image.

4. **`gb_read_palettes`** — For DMG: read I/O registers at 0xFF47 (BGP), 0xFF48 (OBP0), 0xFF49 (OBP1) via `pyboy.memory[]`. For CGB: sequential reads through BCPS/BCPD (0xFF68/0xFF69) and OCPS/OCPD (0xFF6A/0xFF6B). Delegate register parsing to domain functions.

### PyBoy APIs to leverage

- `pyboy.tilemap_background()` / `pyboy.tilemap_window()` — tilemap access with `tile_identifier(col, row)`
- `pyboy.sprite(index)` — sprite properties: `.x`, `.y`, `.tile_identifier`, `.attributes`, `.on_screen`
- `pyboy.memory[address]` — direct I/O register reads for palette data
- PIL Image construction follows the `gb_screenshot` pattern from BLO-022

## Notes

- The spec classifies `gb_inspect_vram`, `gb_get_tilemap`, and `gb_list_sprites` as P1; `gb_read_palettes` as P2. All four are grouped here since they share the same domain module and visual output pattern.
- PyBoy's sprite and tilemap APIs should be verified against the installed version — check `pyboy.sprite()` and `pyboy.tilemap_background()` method signatures.
- CGB palette reading via indexed I/O (BCPS/BCPD) requires careful sequential register access; consider whether to support CGB in V1 or defer to a follow-up ticket.
- Tile rendering assumes standard 2bpp Game Boy tile format (8x8 pixels, 16 bytes per tile).
