# BLO-027: Breakpoint system

## Objective

Implement the breakpoint management tools: `gb_set_breakpoint`, `gb_remove_breakpoint`, and `gb_list_breakpoints`. After this ticket, an LLM can set execution breakpoints at specific addresses (with optional conditions), remove them, and list all active breakpoints. This enables targeted debugging — the LLM can stop execution precisely when a function of interest is called.

## Context

The spec defines three breakpoint tools as P1 in §2:

- **§2 `gb_set_breakpoint`** (P1): Set an execution breakpoint. Input: `address`, `bank` (optional), `condition` (optional, e.g., "A == 0x01"). Output: `breakpoint_id`. PyBoy: `pyboy.hook_register(bank, address, callback)`. Conditions are evaluated in the Python callback by inspecting `register_file`.
- **§2 `gb_remove_breakpoint`** (P1): Remove a breakpoint. Input: `breakpoint_id`. Output: confirmation.
- **§2 `gb_list_breakpoints`** (P1): List active breakpoints. Input: none. Output: list of breakpoints with addresses, banks, and conditions.

The spec's V1 limitation note is important: "execution breakpoints only. Memory read/write watchpoints are not supported by PyBoy and would require polling-based workarounds (deferred to V2)."

`EmulatorSession` (from BLO-021) already has a `breakpoints` dict stub that can be used to track breakpoint state. The `gb_run_until` tool already demonstrates the hook pattern — breakpoints generalize this to persistent hooks that fire during normal execution.

## Acceptance Criteria

- [ ] `gb_set_breakpoint(address, bank?, condition?)` registers a PyBoy hook at the given address and returns a unique `breakpoint_id` (string or int).
- [ ] Breakpoints persist across `gb_step` and `gb_run_until` calls — they are not removed after firing (unlike the one-shot hooks used by `gb_run_until`).
- [ ] Optional `bank` parameter (default: -1, meaning any bank). When specified, the breakpoint only fires in that bank.
- [ ] Optional `condition` parameter supports simple register comparisons (e.g., "A == 0x01", "HL >= 0xC000"). Condition is evaluated in the hook callback; if false, execution continues without stopping.
- [ ] `gb_remove_breakpoint(breakpoint_id)` removes the hook from PyBoy and deletes the breakpoint from the session's tracking dict. Returns `NOT_FOUND` error for unknown IDs.
- [ ] `gb_list_breakpoints()` returns all active breakpoints with their id, address, bank, condition (if any), and hit count.
- [ ] All three tools return `NO_ROM_LOADED` error when no session is active.
- [ ] Domain module (`domain/breakpoints.py`) provides breakpoint validation: address range check, condition parsing/validation. TDD — tests written before implementation.
- [ ] Integration tests cover: set/list/remove lifecycle, conditional breakpoints, error cases (no ROM, invalid ID, invalid condition).

### Hook callback design

The breakpoint callback must:
1. Increment the hit count on the BreakpointInfo.
2. If a condition is set, evaluate it against current registers. If false, return (let execution continue).
3. If no condition or condition is true, signal that execution should pause.

The "pause" mechanism needs design: since PyBoy's tick loop is synchronous, the hook callback could set a flag that `gb_step` and `gb_run_until` check to break out of their tick loops. This is the same pattern `gb_run_until` already uses.

## Notes

- PyBoy's `hook_register(bank, addr, callback, context)` is the underlying mechanism. Bank -1 means "any bank."
- PyBoy's `hook_deregister(addr)` removes all hooks at an address. If multiple breakpoints are set at the same address (different conditions), this is a problem. The implementation may need to handle this by wrapping multiple conditions into a single hook callback per address.
- The spec explicitly scopes V1 to execution breakpoints only: "Memory read/write watchpoints are not supported by PyBoy and would require polling-based workarounds (deferred to V2)."
- Condition evaluation must be safe — no `eval()`. Parse the condition string into a structured representation and evaluate it against register values directly.
- The `get_session_info` tool already reports `breakpoints_count` — ensure this reads from `session.breakpoints` so it stays accurate.
