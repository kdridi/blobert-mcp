# BLO-026: Memory and register write tools

## Objective

Implement `gb_write_memory` and `gb_set_register` tools so that an LLM can modify RAM contents and CPU register values during an emulation session. These write tools enable hypothesis testing — the LLM can inject values into memory or registers and observe the resulting behavior, which is essential for understanding undocumented game logic.

## Context

The project currently supports reading memory (`gb_read_memory`, BLO-013) and reading registers (`gb_get_registers`, BLO-021), but has no write counterparts. The spec defines both write tools as P1 in sections 2 and 3:

- **§2 `gb_write_memory`** (P1): Write to memory. Input: `address`, `data` (hex string or byte array). Output: confirmation. PyBoy: `pyboy.memory[address] = value`.
- **§2 `gb_set_register`** (P1): Write to a register. Input: `register` (string), `value` (int). Output: confirmation. PyBoy: `register_file` properties are writable.

Without these tools, the LLM can only observe — it cannot experiment. Write tools close that gap and unlock workflows like: "set lives to 99 and see if the display routine reads from 0xC0A0" or "set A=0x00 before calling this function to test the zero branch."

## Acceptance Criteria

- [ ] `gb_write_memory(address, data)` writes bytes to memory at the given address. `data` is a hex string (e.g., "FF0042"). Returns confirmation with address and byte count written.
- [ ] `gb_write_memory` validates that `address` is in a writable range (RAM: 0xC000-0xDFFF, HRAM: 0xFF80-0xFFFE, VRAM: 0x8000-0x9FFF, OAM: 0xFE00-0xFE9F). Returns `INVALID_ADDRESS` error for ROM or unmapped regions.
- [ ] `gb_write_memory` validates the hex string (even length, valid hex characters). Returns an error for malformed input.
- [ ] `gb_write_memory` returns `NO_ROM_LOADED` error when no session is active.
- [ ] `gb_set_register(register, value)` writes a value to the specified CPU register. Supported registers: A, B, C, D, E, F, H, L, SP, PC.
- [ ] `gb_set_register` validates register names via `domain/registers.py`. Returns an error for unknown register names.
- [ ] `gb_set_register` validates value ranges (0x00-0xFF for 8-bit registers, 0x0000-0xFFFF for 16-bit SP/PC). Returns an error for out-of-range values.
- [ ] `gb_set_register` returns `NO_ROM_LOADED` error when no session is active.
- [ ] Domain layer (`domain/registers.py`) provides register validation: name normalization, valid name check, value range check. TDD — tests written before implementation.
- [ ] Integration tests cover both tools: successful writes, error cases (bad address, bad register, no ROM).

## Notes

- PyBoy's `memory[address] = value` writes a single byte. For multi-byte writes, loop over the data.
- The F register on the Game Boy only uses the upper 4 bits (flags Z, N, H, C). Writing to F should mask to 0xF0. This is a domain-level concern for `validate_register_value`.
- Composite 16-bit registers (AF, BC, DE, HL) could be supported as a stretch goal but are not required by the spec. The individual 8-bit registers cover all use cases.
- The writable address validation is conservative — it prevents accidental writes to ROM space, which PyBoy would likely ignore anyway. I/O registers (0xFF00-0xFF7F) are intentionally excluded from the writable set for safety, though this could be relaxed later.
