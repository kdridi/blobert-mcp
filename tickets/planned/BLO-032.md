# BLO-032: LCD and hardware register tools

## Objective

Implement four MCP tools that expose hardware I/O register state: `gb_get_lcd_status`, `gb_get_timer_state`, `gb_get_audio_state`, and `gb_get_serial_state`. These are direct memory reads at known I/O register addresses, parsed into structured JSON. A shared domain module (`domain/io_registers.py`) provides register definitions, address constants, and parsing logic.

## Context

The emulator already provides `gb_get_bank_info` and `gb_get_interrupt_status` for hardware state inspection (P0, implemented). The remaining hardware register tools cover LCD/PPU state, timer, audio, and serial — all essential for understanding rendering behavior, timing, and peripheral usage. See `docs/mcp-spec.md` section 7 ("Hardware-specific and edge cases") for full specifications. `gb_get_lcd_status` is P1; `gb_get_timer_state`, `gb_get_audio_state`, and `gb_get_serial_state` are P2.

## Acceptance Criteria

- [ ] `gb_get_lcd_status()` tool implemented — returns LCDC (0xFF40), STAT (0xFF41), SCY (0xFF42), SCX (0xFF43), LY (0xFF44), LYC (0xFF45), WY (0xFF4A), WX (0xFF4B)
- [ ] LCDC and STAT register values are parsed into individual flag fields (e.g., LCDC bit 7 = LCD enable, STAT bits 0-1 = mode)
- [ ] `gb_get_timer_state()` tool implemented — returns DIV (0xFF04), TIMA (0xFF05), TMA (0xFF06), TAC (0xFF07), plus derived `enabled` and `frequency` fields from TAC
- [ ] `gb_get_audio_state(channel?)` tool implemented — returns NRxx registers for specified channel (1-4) or all channels; includes wave RAM (0xFF30-0xFF3F) for channel 3
- [ ] `gb_get_serial_state()` tool implemented — returns SB (0xFF01), SC (0xFF02), plus derived `transfer_in_progress` field
- [ ] Domain module `domain/io_registers.py` created with register address constants, bitfield definitions, and parsing functions
- [ ] All parsing functions are pure (take raw byte values, return structured dicts)
- [ ] Tests written for domain parsing logic (TDD)
- [ ] Tests written for tool handlers
- [ ] Error handling: returns proper error format for NO_ROM_LOADED

## Technical Approach

1. **Domain module** (`domain/io_registers.py`):
   - Register address constants (e.g., `LCDC_ADDR = 0xFF40`, `STAT_ADDR = 0xFF41`, etc.)
   - `parse_lcdc(value: int) -> dict` — decode all 8 LCDC bits (LCD enable, window tilemap, window enable, BG/window tile data, BG tilemap, OBJ size, OBJ enable, BG/window priority)
   - `parse_stat(value: int) -> dict` — decode STAT bits (LYC=LY coincidence interrupt, mode 2/1/0 OAM interrupt, mode flag)
   - `parse_lcd_status(lcdc, stat, scy, scx, ly, lyc, wy, wx) -> dict` — aggregate LCD state
   - `parse_tac(value: int) -> dict` — decode TAC (enabled flag, frequency from bits 0-1)
   - `parse_timer_state(div, tima, tma, tac) -> dict` — aggregate timer state
   - `parse_audio_state(registers: dict, channel: int | None) -> dict` — aggregate audio registers
   - `parse_serial_state(sb, sc) -> dict` — decode SC (transfer in progress, clock source)

2. **Tools** (`tools/io_registers.py`):
   - Each tool reads the relevant I/O register addresses via `pyboy.memory[addr]`
   - Delegates parsing to domain functions
   - Returns structured JSON

3. **Tests** (TDD — write before implementation):
   - `tests/domain/test_io_registers.py` — parsing edge cases for each register group
   - `tests/tools/test_io_registers.py` — tool handler behavior

## Notes

- All four tools are simple I/O register reads — no emulator state mutation, no complex control flow.
- `gb_get_lcd_status` is P1 (important for understanding rendering). The other three are P2 but grouped here because they share the same domain module and implementation pattern.
- Audio registers span NR10-NR52 (0xFF10-0xFF26) plus wave RAM (0xFF30-0xFF3F). The `channel` parameter allows querying a single channel or all.
- TAC frequency mapping: bits 0-1 map to 4096 Hz, 262144 Hz, 65536 Hz, 16384 Hz.
