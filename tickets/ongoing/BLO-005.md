# BLO-005: Knowledge base with SQLite storage

## Objective

Implement the P0 knowledge base tools backed by SQLite, enabling the LLM to annotate addresses, define functions and variables, and search its accumulated knowledge. After this ticket, every discovery the LLM makes during ROM exploration can be persisted and queried — turning ephemeral observations into a growing understanding of the ROM.

## Context

The knowledge base is what makes blobert-mcp more than a debugger — it's the LLM's memory. Without it, every exploration session starts from zero. The KB tools are used throughout the example workflow (step 10: `kb_annotate`) and are referenced by other tools (the disassembler enriches output with KB labels).

Covers spec §8: `kb_annotate`, `kb_define_function`, `kb_define_variable`, `kb_search` (all P0).

Spec recommends SQLite for the KB: "Relational queries, referential integrity, history via versioning table, single file, portable."

## Acceptance Criteria

- [x] SQLite database is created per ROM (named after ROM filename, stored alongside or in a configurable location).
- [x] `kb_annotate` — Creates an annotation for an address with label, type (code/data/gfx/audio/text), and comment. Returns annotation_id. Updates existing annotation if address+bank already annotated.
- [x] `kb_define_function` — Defines a function with address, end_address, bank, name, params, description, returns. Returns function_id.
- [x] `kb_define_variable` — Names a RAM variable with address, name, type (u8/u16/bool/enum), description. Returns variable_id.
- [x] `kb_search` — Searches annotations by query text. Supports filters: label, comment, address, type. Returns max 50 results sorted by relevance. Exact match on addresses, substring match on labels/comments.
- [x] Database schema has tables: `annotations`, `functions`, `variables` with appropriate columns and indices.
- [x] KB is accessible from other tools (disassembler can look up labels for addresses).
- [x] KB persists across server restarts (SQLite file on disk).
- [x] All tools return proper errors for invalid inputs (e.g., invalid type enum, malformed address).
- [x] Unit tests cover CRUD operations and search functionality.

## Technical Approach

1. **`src/blobert_mcp/kb/database.py`** — KnowledgeBase class:
   - `__init__(db_path)` — opens/creates SQLite database, runs schema migration
   - `annotate(address, bank, label, type, comment) -> annotation_id`
   - `define_function(address, end_address, bank, name, params, description, returns) -> function_id`
   - `define_variable(address, name, type, description) -> variable_id`
   - `search(query, filter) -> list[dict]`
   - `get_label(address, bank=None) -> str | None` — fast lookup for disassembler enrichment
   - Uses `sqlite3` stdlib module (no extra dependency)

2. **Database schema:**
   ```sql
   CREATE TABLE annotations (
       id INTEGER PRIMARY KEY,
       address INTEGER NOT NULL,
       bank INTEGER,
       label TEXT,
       type TEXT CHECK(type IN ('code','data','gfx','audio','text')),
       comment TEXT,
       created_at TEXT DEFAULT CURRENT_TIMESTAMP,
       updated_at TEXT DEFAULT CURRENT_TIMESTAMP,
       UNIQUE(address, bank)
   );

   CREATE TABLE functions (
       id INTEGER PRIMARY KEY,
       address INTEGER NOT NULL,
       end_address INTEGER,
       bank INTEGER,
       name TEXT NOT NULL,
       params TEXT,  -- JSON array
       description TEXT,
       returns TEXT,
       created_at TEXT DEFAULT CURRENT_TIMESTAMP,
       UNIQUE(address, bank)
   );

   CREATE TABLE variables (
       id INTEGER PRIMARY KEY,
       address INTEGER NOT NULL,
       name TEXT NOT NULL,
       type TEXT CHECK(type IN ('u8','u16','bool','enum')),
       description TEXT,
       created_at TEXT DEFAULT CURRENT_TIMESTAMP,
       UNIQUE(address)
   );
   ```

3. **`src/blobert_mcp/tools/kb.py`** — MCP tool wrappers that connect KB operations to the MCP server.

4. **DB file location** — Default: `~/.blobert-mcp/<rom_name>.db`. Created when first KB operation is performed on a loaded ROM.

5. **Integration with EmulatorSession** — The `EmulatorSession` holds a reference to the active `KnowledgeBase` instance. When a ROM is loaded, the corresponding KB database is opened (or created). When the ROM is unloaded, the KB connection is closed.

## Notes

- SQLite is in the Python stdlib — no extra dependency needed.
- The `get_label()` method is a hot path (called for every instruction in disassembly). Consider caching or an in-memory index for addresses with labels.
- `kb_search` relevance: for V1, substring matching with simple ranking (exact match > prefix match > substring) is sufficient. Full-text search (FTS5) can be added later.
- The `params` field in `functions` table stores a JSON array since function parameters have variable structure.
- UPSERT behavior for `kb_annotate`: if an annotation exists for the same address+bank, update it rather than creating a duplicate. Use `INSERT ... ON CONFLICT(address, bank) DO UPDATE`.
- The KB is per-ROM. If the user loads a different ROM, a different KB database is used.
