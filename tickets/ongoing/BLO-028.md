# BLO-028: Save state management tools

## Objective

Implement `gb_list_states` and `gb_delete_state` MCP tools to give the LLM full control over saved emulator states. After this ticket, the LLM can enumerate existing save states with metadata and remove states that are no longer needed, completing the save state lifecycle defined in spec section 4.

## Context

Save states are stored in-memory as a `BytesIO` dict in `EmulatorSession`. The `gb_save_state` and `gb_load_state` tools already exist (P0, completed in BLO-021), but there is no way to list or delete states. The spec (section 4) defines `gb_list_states` as P1 and `gb_delete_state` as P2. Both are simple dict operations on the existing save state storage.

## Acceptance Criteria

- [ ] `gb_list_states()` tool is registered and returns a list of all saved states with metadata: `state_id`, `name`, `frame_count`, `PC`, and `timestamp`
- [ ] `gb_delete_state(state_id)` tool is registered and removes the specified state from the session
- [ ] `gb_delete_state` returns an appropriate error when given a non-existent `state_id`
- [ ] `get_session_info` reflects the correct `save_states_count` after deletions
- [ ] Unit tests cover both tools (list empty, list with states, delete existing, delete non-existent)
- [ ] Tools follow the existing error format: `{"error": "CODE", "message": "..."}`

## Technical Approach

1. **Metadata storage**: Extend the save state dict entries (or add a parallel metadata dict) to store `name`, `frame_count`, `PC`, and `timestamp` at save time. This may require a minor update to `gb_save_state` to capture metadata.
2. **`gb_list_states` tool**: Iterate `EmulatorSession.save_states`, return metadata for each entry. No emulator interaction needed â€” pure dict read.
3. **`gb_delete_state` tool**: Pop the state from the dict. Return confirmation or `NOT_FOUND` error.
4. **Tool registration**: Add both tools to the MCP server tool registry following the existing pattern.
5. **Tests**: Unit tests for list and delete operations, including edge cases.

## Notes

- The save state storage is currently a simple dict keyed by `state_id`. Metadata (frame_count, PC) should be captured at save time so `gb_list_states` doesn't need to load each state to inspect it.
- `gb_delete_state` is classified as P2 in the spec but is trivial to implement alongside `gb_list_states` (P1), so both are bundled here.
