# BLO-041: KB function info and statistics

## Objective

Implement two knowledge-base query tools: `kb_get_function_info` for retrieving comprehensive information about a function, and `kb_stats` for reporting decompilation progress statistics. Both tools aggregate data from existing SQLite tables. This ticket is done when both tools are registered, tested, and return correct results against a populated knowledge base.

## Context

The knowledge base (implemented in BLO-005) stores annotations, functions, and variables in SQLite. The spec (section 8) defines `kb_get_function_info` (P1) and `kb_stats` (P2) as query tools that aggregate existing data. With 25 P0 tools already implemented, these aggregation tools add analytical capability without requiring new storage infrastructure.

See `docs/mcp-spec.md` section 8 for tool definitions.

### ROM vs non-ROM addresses

The KB accepts annotations at any address in `[0x0000, 0xFFFF]` (`validate_address` in `domain/kb.py`), but the Game Boy memory map is split:

| Range | Region | ROM? |
|---|---|---|
| `0x0000-0x3FFF` | ROM bank 0 | Yes |
| `0x4000-0x7FFF` | ROM bank N (switchable) | Yes |
| `0x8000-0x9FFF` | VRAM | No |
| `0xA000-0xBFFF` | External RAM | No |
| `0xC000-0xDFFF` | Work RAM | No |
| `0xFF00+` | I/O, HRAM | No |

`kb_define_variable` is explicitly for RAM addresses. `kb_annotate` and `kb_define_function` can target any address. The `kb_stats` coverage calculation must account for this: the numerator (`annotated`) and denominator (`total_addresses`) must both be ROM-only. Mixing ROM and non-ROM annotations in the numerator against a ROM-only denominator would give a misleading or >100% coverage percentage.

**ROM address boundary**: `address < 0x8000`. Annotations and functions at `address >= 0x8000` are valid KB entries but are excluded from the ROM coverage calculation.

## Acceptance Criteria

- [ ] `kb_get_function_info(name_or_address)` tool is registered and callable via MCP
- [ ] `kb_get_function_info` returns function definition, local variables, cross-references, and comments for the given function (by name or address lookup)
- [ ] `kb_get_function_info` returns a clear error when the function is not found
- [ ] `kb_stats()` tool is registered and callable via MCP
- [ ] `kb_stats` returns JSON with: `total_addresses`, `annotated`, `functions_named`, `variables_named`, `coverage_pct`
- [ ] `annotated` counts only unique ROM-range `(address, bank)` pairs from the annotations table where `address < 0x8000`
- [ ] `coverage_pct` = `annotated / total_addresses * 100`, where both numerator and denominator are ROM-only
- [ ] `total_addresses` = `bank_count * 0x4000` (computed from ROM header byte at `0x0148` via `bank_info.calculate_bank_count`)
- [ ] Domain logic is in a dedicated domain module (pure functions, no tool/MCP imports)
- [ ] Unit tests exist for domain logic (TDD), including a test that verifies non-ROM annotations (`address >= 0x8000`) are excluded from the `annotated` count and `coverage_pct`
- [ ] Tool handler tests exist for both tools

## Technical Approach

1. **Domain module**: Add query functions to the existing KB domain module (or create a new `domain/kb_queries.py`) that accept a database connection and return structured data.
2. **`kb_get_function_info`**: SQL query joining the functions table with annotations (for comments at the function's address range) and variables (for locals within the function's address range). Accept either a name (string match) or address (integer match).
3. **`kb_stats`**: SQL aggregation queries on annotations, functions, and variables tables.
   - `annotated`: `SELECT COUNT(*) FROM annotations WHERE address < 0x8000` (ROM-only).
   - `functions_named`: `SELECT COUNT(*) FROM functions` (all functions, regardless of address).
   - `variables_named`: `SELECT COUNT(*) FROM variables` (all variables — these are typically RAM).
   - `total_addresses`: computed by the tool handler from `bank_info.calculate_bank_count(session.pyboy.memory[0x0148]) * 0x4000`.
   - `coverage_pct`: pure domain function that takes `annotated` and `total_addresses` as inputs, returns `annotated / total_addresses * 100`. This keeps the domain layer free of session/hardware dependencies.
4. **Tool handlers**: Thin wrappers that validate input, call domain functions, and format output. Register as MCP tools. The `kb_stats` handler reads the ROM header byte, computes `total_addresses`, queries the KB for counts, and passes all values to the domain function.
5. **Tests**: Write domain tests first (TDD), then tool handler tests.

## Notes

- Both tools are primarily SQL queries on existing tables — no schema changes needed.
- `kb_get_function_info` should handle lookup by both name (string) and address (integer/hex string).
- Coverage percentage in `kb_stats` depends on knowing the total ROM size, which comes from the loaded ROM header byte at `0x0148`.
- The ROM address boundary (`0x8000`) is a fundamental Game Boy hardware constant, not a magic number. Define it as a named constant in the domain module (e.g., `ROM_ADDRESS_LIMIT = 0x8000`).
