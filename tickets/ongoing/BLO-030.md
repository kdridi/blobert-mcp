# BLO-030: Memory diff between save states

## Objective

Implement `gb_memory_diff` MCP tool that compares memory contents between two save states and returns the list of changed bytes. After this ticket, the LLM can isolate which memory locations were modified by a game action (e.g., pressing a button, advancing frames), which is essential for identifying game variables and data structures.

## Context

The spec (section 2, `gb_memory_diff`, and section 3 as an alternative to `gb_get_memory_access_log`) defines this as P1. Save states already exist in-memory (BLO-021). The tool loads each state, dumps the requested memory region, restores the original state, and diffs the two dumps.

The existing `domain/memory_map.py` module defines memory regions (WRAM, HRAM, VRAM, etc.) and can be used to filter diffs by region name instead of requiring raw address/length parameters.

## Acceptance Criteria

- [ ] `gb_memory_diff(state_id_a, state_id_b, regions?)` tool is registered
- [ ] The tool compares memory between two save states and returns a list of `(address, old_value, new_value)` changes
- [ ] Optional `regions` parameter accepts a list of region names (e.g., `["WRAM", "HRAM"]`) to filter the comparison using `domain/memory_map.py`
- [ ] When no `regions` parameter is provided, a sensible default is used (e.g., WRAM + HRAM)
- [ ] The original emulator state is restored after the comparison (neither state_a nor state_b remains loaded)
- [ ] Returns appropriate errors for non-existent state IDs
- [ ] Output respects the truncation convention: includes `"truncated": true` and `"total": N` when results exceed the limit
- [ ] Unit tests cover: identical states (no diff), states with changes, region filtering, non-existent state IDs, truncation
- [ ] Tool follows the existing error format: `{"error": "CODE", "message": "..."}`

## Technical Approach

1. **Domain logic** (can be inline or a small domain module):
   - `diff_memory(bytes_a, bytes_b, base_address) -> list[MemoryChange]`
   - Pure function: compare two byte sequences, return list of `(address, old, new)` for each differing byte
   - No emulator dependency — operates on raw bytes

2. **Tool handler**:
   - Save current state to a temporary buffer
   - Load `state_a`, dump memory for requested regions
   - Load `state_b`, dump memory for requested regions
   - Restore original state from temporary buffer
   - Call domain diff function on the two dumps
   - Apply truncation limit and format output

3. **Region resolution**: Use `domain/memory_map.py` to convert region names to `(start_address, length)` pairs.

4. **Tests**: Synthetic byte arrays for domain diff logic. Integration-level tests for the tool handler if emulator test fixtures are available.

## Notes

- The spec suggests `address` + `length` as inputs, but region-based filtering is more ergonomic for the LLM. Supporting both (raw address/length OR named regions) would be ideal.
- Default regions should exclude VRAM (0x8000-0x9FFF) and I/O (0xFF00-0xFF7F) to reduce noise — game state is typically in WRAM and HRAM.
- Performance consideration: WRAM is 8KB (0xC000-0xDFFF), HRAM is 127 bytes (0xFF80-0xFFFE). Diffing these is trivial.
- This tool serves as the V1 alternative to `gb_get_memory_access_log` (P2/deferred), as noted in spec section 3.
