# BLO-003: SM83 disassembler engine

## Objective

Implement a complete SM83 (Game Boy CPU) disassembler and expose it through the three P0 disassembly MCP tools. After this ticket, an LLM can disassemble code at any address, view code around the current PC, and trace a function from entry to return — the core analysis capability.

## Context

The disassembler is the single most important analysis tool. It translates raw bytes into human-readable assembly, which the LLM then interprets. It must handle the full SM83 instruction set (256 base opcodes + 256 CB-prefixed opcodes), including correct operand decoding for immediate values, relative jumps, and 16-bit addresses.

Covers spec §1: `disassemble_range`, `disassemble_function`, `disassemble_at_pc`.

## Acceptance Criteria

- [ ] SM83 instruction decoder handles all 256 base opcodes and 256 CB-prefixed opcodes.
- [ ] Each decoded instruction includes: address, raw hex bytes, mnemonic, operands, size (1-3 bytes).
- [ ] `disassemble_range` — Disassembles from `address` for `length` bytes or until `end_address`. Supports optional `bank`. Returns max 256 instructions. Returns list of instruction objects.
- [ ] `disassemble_function` — Disassembles from `entry_point` until unconditional RET, JP, or safety limit (max 1024 instructions). Reports function size in bytes.
- [ ] `disassemble_at_pc` — Reads current PC from registers, disassembles `before` instructions before and `after` instructions after. Marks the current PC instruction. Default: before=5, after=20.
- [ ] Relative jump targets (JR) are resolved to absolute addresses and shown as comments.
- [ ] Knowledge base labels are included in output when available (e.g., `CALL 0x1234 ; play_sound`).
- [ ] Disassembler correctly handles instruction boundaries (no mid-instruction starts).
- [ ] Unit tests cover: all instruction categories (loads, arithmetic, jumps, CB-prefix rotates/bits), edge cases (JP to same address, consecutive RET), operand formats (d8, d16, a16, r8 signed offset).

## Technical Approach

1. **`src/blobert_mcp/disasm/opcodes.py`** — SM83 opcode tables:
   - Base table: 256 entries, each with (mnemonic, operand_types, size, flags_affected)
   - CB-prefix table: 256 entries for bit operations
   - Use a data-driven approach (dict/list), not a giant switch statement

2. **`src/blobert_mcp/disasm/decoder.py`** — Instruction decoder:
   - `decode_instruction(bytes, address) -> Instruction` — decode one instruction
   - `Instruction` dataclass: address, raw_bytes, mnemonic, operands, size
   - Handle immediate values: d8 (unsigned byte), d16 (little-endian word), r8 (signed offset for JR), a16 (absolute address)

3. **`src/blobert_mcp/disasm/disassembler.py`** — High-level disassembly functions:
   - `disassemble_range(memory_reader, address, length, bank=None)`
   - `disassemble_function(memory_reader, entry_point, bank=None)`
   - `disassemble_at_pc(memory_reader, pc, before=5, after=20)`
   - The `memory_reader` is a callable that abstracts ROM/memory access

4. **`src/blobert_mcp/tools/disasm.py`** — MCP tool wrappers that connect the disassembler to the MCP server and EmulatorSession.

5. **`disassemble_at_pc` backward scanning** — Scanning backward in a variable-length ISA is ambiguous. Approach: start disassembling from a safe distance before PC (e.g., `before * 3` bytes back) and find the instruction sequence that aligns with PC. If alignment fails, fall back to starting from PC minus a fixed offset and flag potential misalignment.

## Notes

- The SM83 is a modified Z80. Key differences: no IX/IY registers, no IN/OUT instructions, STOP and SWAP are SM83-specific, CB prefix works differently.
- Reference: [SM83 opcode table](https://gbdev.io/gb-opcodes/optables/) — use this as the canonical source.
- The disassembler is a pure logic module with no PyBoy dependency — it just needs a byte source. This makes it independently testable.
- `disassemble_function` endpoint detection: stop at unconditional `RET` (0xC9), `RETI` (0xD9), unconditional `JP` (0xC3) that jumps outside the function. Do NOT stop at conditional returns/jumps.
- Performance: the disassembler will be called frequently. Keep it fast — avoid unnecessary allocations.
