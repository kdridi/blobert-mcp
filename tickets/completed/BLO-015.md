# BLO-015: SM83 opcode table and instruction decoder

## Objective

Implement the complete SM83 opcode tables (256 base + 256 CB-prefixed) and the instruction decoder as a pure domain module. After this ticket, any caller with a byte source can decode any single SM83 instruction into a structured `Instruction` dataclass, correctly handling all operand formats and resolving relative jump targets to absolute addresses.

## Context

This is the decoder foundation that BLO-016 (disassembly algorithms and MCP tools) builds upon. The opcode tables are a large, error-prone data-entry task — 512 entries — that must be independently verified via TDD before higher-level algorithms are written on top of them. Isolating this work ensures the decoder is rock-solid before any algorithm depends on it.

Covers the decoder layer for spec §1: `disassemble_range`, `disassemble_function`, `disassemble_at_pc`.

## Acceptance Criteria

- [x] Opcode table covers all 256 base opcodes: each entry has mnemonic, operand_types, and size (1–3 bytes).
- [x] CB-prefix table covers all 256 CB opcodes: each entry has mnemonic, operand_types, and size (always 2 bytes).
- [x] `decode_instruction(data: bytes, address: int) -> Instruction` decodes any single SM83 instruction. `Instruction` dataclass fields: `address`, `raw_bytes`, `mnemonic`, `operands`, `size`.
- [x] Relative jump targets (JR, JR cc) are resolved to absolute addresses and included in operands. Instruction boundaries are always respected (no mid-instruction decoding).
- [x] Unit tests (written before implementation — TDD) cover: all instruction categories (8-bit loads, 16-bit loads, arithmetic/logic, jumps, calls/returns, CB rotates/shifts/bit ops/swaps), all operand formats (d8, d16, a16, r8 signed offset), and edge cases (CB prefix dispatch, STOP, HALT, undefined/illegal opcodes).

## Technical Approach

1. **`src/blobert_mcp/domain/disasm/__init__.py`** — empty, marks the package.

2. **`src/blobert_mcp/domain/disasm/opcodes.py`** — two lookup structures:
   - `BASE_OPCODES: dict[int, OpcodeEntry]` — keys 0x00–0xFF
   - `CB_OPCODES: dict[int, OpcodeEntry]`  — keys 0x00–0xFF
   - `OpcodeEntry` namedtuple or dataclass: `(mnemonic: str, operand_types: tuple[str, ...], size: int)`
   - Operand type tokens: `"d8"`, `"d16"`, `"a16"`, `"r8"`, `"r8s"` (signed), `"(a8)"`, `"(a16)"`, `"(C)"`, register names as literals
   - Data-driven: no switch statements, just table lookups

3. **`src/blobert_mcp/domain/disasm/decoder.py`** — instruction decoder:
   - `Instruction` dataclass: `address: int`, `raw_bytes: bytes`, `mnemonic: str`, `operands: list[str]`, `size: int`
   - `decode_instruction(data: bytes, address: int) -> Instruction` — looks up opcode in table, reads operand bytes, formats operand strings (hex with `0x` prefix, JR resolved to absolute)
   - No project imports. No PyBoy. stdlib + dataclasses only.

4. **TDD sequence** — mandatory order:
   - Write `tests/domain/disasm/test_opcodes.py` first: verify table completeness (all 256+256 entries present), spot-check known opcodes per category
   - Write `tests/domain/disasm/test_decoder.py`: one test per operand format, one test per instruction category
   - Implement `opcodes.py` until opcode tests pass
   - Implement `decoder.py` until decoder tests pass

## Notes

- The SM83 is a modified Z80. Key differences: no IX/IY registers, no IN/OUT, STOP and SWAP are SM83-specific, CB prefix works differently from Z80.
- Reference: https://gbdev.io/gb-opcodes/optables/ — canonical opcode source.
- Undefined opcodes (0xD3, 0xDB, 0xDD, 0xE3, 0xE4, 0xEB, 0xEC, 0xED, 0xF4, 0xFC, 0xFD on base table) should decode as `DB 0xXX` (data byte) with size 1 — do not raise exceptions.
- All CB opcodes are exactly 2 bytes (0xCB prefix + opcode byte). Size is always 2.
- JR resolution: `absolute = (address + 2 + signed_offset) & 0xFFFF` — wraps at 16-bit boundary.
