# BLO-023: Backfill timestamps on existing tickets from git history

## Objective

Update all existing tickets to use the `YYYY-MM-DD HH:MM:SS` datetime format (D-016) by extracting precise timestamps from git history. After this ticket, every ticket file — backlog, completed, and rejected — has accurate datetime fields and log entries, enabling full temporal sorting across the project.

## Context

D-016 introduced the `YYYY-MM-DD HH:MM:SS` format for new tickets, but left existing tickets with date-only values to avoid fabricating data. Git history contains the precise commit timestamps for every ticket event (creation, activation, completion, rejection). This ticket uses that data to backfill accurate datetimes.

## Acceptance Criteria

- [x] All ticket frontmatter `created:` fields updated to `YYYY-MM-DD HH:MM:SS` using the timestamp of the commit that first added the file.
- [x] All ticket frontmatter `updated:` fields updated to `YYYY-MM-DD HH:MM:SS` using the timestamp of the last commit that modified the file.
- [x] All log entries (`- YYYY-MM-DD: ...`) updated to `- YYYY-MM-DD HH:MM:SS: ...` using the timestamp of the commit that introduced each log entry.
- [x] Timestamps are sourced exclusively from git history — no fabricated data.
- [x] BLO-020, BLO-021, BLO-022 (already in new format) are not modified.
- [x] All 23 existing tickets updated: 15 completed, 3 rejected, 2 backlog (BLO-017, BLO-023 itself).

## Technical Approach

### Data source: git log

Each ticket event has a corresponding commit with a precise timestamp:

```bash
# Find when a ticket file was first added
git log --diff-filter=A --format="%ai" -- tickets/**/BLO-XXX.md

# Find when a ticket file was last modified
git log -1 --format="%ai" -- tickets/**/BLO-XXX.md

# Find the commit that introduced a specific log entry
git log -S "Ticket activated" --format="%ai" -- tickets/**/BLO-XXX.md
```

### Alternative methods

If git log cannot resolve a specific log entry's timestamp (e.g., multiple log entries added in the same commit), use `git blame` to pinpoint which commit introduced each line:

```bash
git blame --porcelain tickets/completed/BLO-XXX.md
```

### Workflow

1. For each ticket file, query git for creation and last-modified timestamps.
2. For each log entry, query git for the commit that introduced it (via `git log -S` or `git blame`).
3. Update frontmatter `created:` and `updated:` fields.
4. Update log entry date prefixes.
5. Verify no data was fabricated — every timestamp traces to a git commit.

## Notes

- Git timestamps include timezone offsets (e.g., `+0100`, `+0200`). Normalize to local time (matching the format used by BLO-020/021/022).
- Some tickets had multiple log entries added in a single commit (e.g., "Ticket created." and "Ticket activated." when a ticket was created and immediately activated). In these cases, use the same commit timestamp for both entries.
- BLO-004's `updated` field was already set to `2026-03-29 14:52:30` during its rejection. Its log entries from before that date retain date-only format until this ticket updates them.
- This ticket only modifies files in `tickets/` — no code changes.
