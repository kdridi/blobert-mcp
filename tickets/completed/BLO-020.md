# BLO-020: Domain layer for dynamic tools (TDD)

## Objective

Implement the pure domain modules needed by the dynamic execution tools — register formatting and button validation — using strict TDD. After this ticket, the domain layer provides tested, reusable functions for composing CPU register output and validating Game Boy button inputs, ready for the tool layer to consume.

## Context

This is the first of three tickets decomposed from BLO-004 (rejected). Following D-008 (decompose by architectural layer), the domain layer is implemented and tested before any tool code is written.

Register formatting is reused by 4+ tools (`gb_get_registers`, `gb_step`, `gb_run_until`, `gb_load_state`). Button validation is used by `gb_press_button`. Implementing these as pure domain functions enables TDD with no emulator dependency.

## Acceptance Criteria

- [x] `domain/registers.py` — `format_registers(a, b, c, d, e, f, h, l, sp, pc) -> dict` takes 10 individual register values (all ints) and returns the spec-compliant JSON structure: `{A, B, C, D, E, F, H, L, AF, BC, DE, HL, SP, PC, flags: {Z, N, H, C}}`.
- [x] Composite registers computed correctly: AF = (A << 8) | F, BC = (B << 8) | C, DE = (D << 8) | E, HL = (H << 8) | L.
- [x] Flags extracted from F register: Z = bit 7, N = bit 6, H = bit 5, C = bit 4. Returned as booleans.
- [x] All register values formatted as hex strings (e.g., `"0xAB"` for 8-bit, `"0xABCD"` for 16-bit).
- [x] `domain/buttons.py` — `VALID_BUTTONS` set: {a, b, start, select, up, down, left, right}. `VALID_ACTIONS` set: {press, release}.
- [x] `domain/buttons.py` — `validate_button(name: str) -> str` returns the normalized lowercase button name or raises `ValueError` for invalid buttons.
- [x] `domain/buttons.py` — `validate_action(action: str) -> str` returns the normalized lowercase action name or raises `ValueError` for invalid actions.
- [x] All tests written BEFORE implementation (TDD). Tests use synthetic int inputs only — no PyBoy dependency.
- [x] Domain modules import nothing from the project — stdlib only.

## Technical Approach

### TDD workflow (mandatory)

1. Create `tests/domain/test_registers.py` — write tests first
2. Create `src/blobert_mcp/domain/registers.py` — implement until tests pass
3. Create `tests/domain/test_buttons.py` — write tests first
4. Create `src/blobert_mcp/domain/buttons.py` — implement until tests pass

### `domain/registers.py`

Pure function that takes 10 ints and returns a structured dict:

```python
def format_registers(a: int, b: int, c: int, d: int, e: int,
                     f: int, h: int, l: int, sp: int, pc: int) -> dict:
    return {
        "A": f"0x{a:02X}", "B": f"0x{b:02X}", ...,
        "AF": f"0x{(a << 8) | f:04X}", ...,
        "SP": f"0x{sp:04X}", "PC": f"0x{pc:04X}",
        "flags": {
            "Z": bool(f & 0x80), "N": bool(f & 0x40),
            "H": bool(f & 0x20), "C": bool(f & 0x10),
        },
    }
```

### `domain/buttons.py`

Constants and validation, following the pattern in `domain/kb.py`:

```python
VALID_BUTTONS = {"a", "b", "start", "select", "up", "down", "left", "right"}
VALID_ACTIONS = {"press", "release"}

def validate_button(name: str) -> str: ...
def validate_action(action: str) -> str: ...
```

## Notes

- The spec (§2, `gb_get_registers`) defines the exact JSON structure: `{A, B, C, D, E, F, H, L, AF, BC, DE, HL, SP, PC, flags: {Z, N, H, C}}`.
- On the Game Boy, only the upper 4 bits of F are used. The lower 4 bits are always 0. Tests should verify this doesn't affect flag extraction.
- `validate_button` should be case-insensitive (accept "Start" → "start").
- This ticket mirrors BLO-012 (domain TDD for static tools) in structure and approach.
