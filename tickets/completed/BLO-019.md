# BLO-019: Design decision log (docs/decisions.md)

## Objective

Create a single-file decision log at `docs/decisions.md` with backfilled decisions from the project's history, update the ticket template to cross-reference decisions, and add CLAUDE.md guidance. After this ticket, every significant design choice has a stable `D-XXX` identifier that can be referenced from tickets, code, and docs.

## Context

Design decisions are currently scattered across `docs/journal.md` (8-row table), rejected ticket logs (BLO-002, BLO-003), completed ticket notes (BLO-005), and `docs/architecture.md` (inline rules). There is no central index or stable identifiers. When a future conversation needs to understand "why SQLite?" or "why domain isolation?", the AI must search multiple files.

A single `docs/decisions.md` with an index table and lightweight `D-XXX` entries provides:
- One-read discovery for the AI at session start
- Stable cross-reference IDs for tickets and docs
- Minimal maintenance burden (append-only)

## Acceptance Criteria

- [x] `docs/decisions.md` exists with an index table and ~12-15 backfilled decisions
- [x] Each decision has: ID (D-XXX), date, ticket reference, context, decision, and rationale
- [x] `tickets/TEMPLATE.md` has a `## Decisions` section for cross-referencing D-XXX IDs
- [x] `CLAUDE.md` has guidance on when and how to record decisions
- [x] `docs/architecture.md` references `docs/decisions.md` for rationale behind its rules
- [x] Existing decisions from journal.md, rejected tickets, and BLO-005/BLO-011 are captured

## Technical Approach

1. Create `docs/decisions.md` with index table + individual entries
2. Backfill from: journal.md "Decisions Made" table, BLO-002 rejection, BLO-003 rejection, BLO-005 notes, BLO-011 architecture choices, BLO-018 pre-commit decision, BLO-005 implementation (NULL bank sentinel, label cache, search ranking)
3. Add `## Decisions` section to ticket template
4. Add decisions guidance to CLAUDE.md Architecture section
5. Add a reference line in architecture.md

## Notes

- Format is deliberately lightweight: Context / Decision / Why — not MADR-heavy (7-10 sections).
- Single file preferred over individual ADR files because the primary consumer is an AI that reads one file per session start.
- The decision log is append-only. Superseded decisions are marked with status, not deleted.
