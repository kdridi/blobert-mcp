# BLO-040: KB symbol import

## Objective

Implement the `kb_import_symbols` tool to import symbol definitions from external files into the knowledge base. Done when the tool supports "sym" (RGBDS .sym files), "pokered" (pokered/pokecrystal .sym format), and "auto" (format auto-detection) modes, with domain parsing logic in a dedicated module and full test coverage.

## Context

Many Game Boy ROMs have existing symbol files from partial disassembly efforts or community projects (pokered, pokecrystal, etc.). Importing these symbols bootstraps the knowledge base with hundreds or thousands of labels, function names, and address annotations in seconds rather than requiring the LLM to discover them one by one. This dramatically accelerates the decompilation workflow for ROMs with existing community knowledge. See `docs/mcp-spec.md` section 8 for the `kb_import_symbols` specification.

## Acceptance Criteria

- [x] `kb_import_symbols(file_path, format?)` tool implemented, accepting format as "sym", "pokered", or "auto" (default: "auto")
- [x] "sym" parser: correctly parses standard RGBDS .sym files (`BB:XXXX label` lines, ignoring `;` comments and blank lines)
- [x] "pokered" parser: correctly parses pokered/pokecrystal .sym format (handles section-based organization and `.`-delimited hierarchical labels)
- [x] "auto" mode: detects the format by inspecting file content (e.g., presence of section headers, label conventions)
- [x] Imported symbols create annotations (with type "code" or "data" inferred from address range) and function definitions (for code labels)
- [x] Returns count of imported symbols: `{"imported": N, "skipped": M, "errors": E}`
- [x] Duplicate labels (already in KB) are skipped, not overwritten, with a count in the response
- [x] Invalid file path returns a clear error
- [x] Domain module `domain/kb_import.py` contains all parsing logic (pure functions, no I/O beyond file reading)
- [x] All domain logic is TDD — tests written before implementation
- [x] Tool handler is thin: validates input, reads file, delegates to domain parser, calls KB methods to store results

## Technical Approach

1. **Domain module** — Create `src/blobert_mcp/domain/kb_import.py` with pure functions:
   - `parse_sym(content: str)` -> list of Symbol(address, bank, label, symbol_type)
   - `parse_pokered(content: str)` -> list of Symbol
   - `detect_format(content: str)` -> "sym" | "pokered"
   - Symbol dataclass: address (int), bank (int | None), label (str), symbol_type ("code" | "data")
2. **Format detection heuristics**:
   - pokered format: lines with `.` in labels (e.g., `VBlank.done`), section names as labels
   - Standard sym: simple `BB:XXXX label` without hierarchical naming
   - Fallback to "sym" if ambiguous
3. **Address range classification**:
   - 0x0000-0x7FFF (ROM): infer "code" type
   - 0x8000-0x9FFF (VRAM): infer "data" type (gfx)
   - 0xA000-0xBFFF (SRAM): infer "data" type
   - 0xC000-0xDFFF (WRAM): infer "data" type
   - 0xFF00-0xFF7F (I/O): infer "data" type
   - 0xFF80-0xFFFE (HRAM): infer "data" type
4. **KnowledgeBase method** — Add `import_symbols(symbols)` method that iterates parsed symbols, checks for duplicates, and calls existing `annotate()` / `define_function()` methods
5. **Tool handler** — Register `kb_import_symbols` MCP tool, read file from disk, delegate parsing to domain, call KB import, return summary

## Notes

- File reading from disk introduces a security consideration: the tool should validate that the file path is within a reasonable scope (e.g., the project directory or a configured import directory). This may warrant a design decision entry.
- The pokered .sym format can contain thousands of symbols. Consider batch insertion for performance.
- ROM bank 0 symbols (0x0000-0x3FFF) are in bank 0; symbols in 0x4000-0x7FFF need the bank number from the sym file.
- Some .sym files include local labels (prefixed with `.`) that are relative to the preceding global label. The parser should handle these by constructing fully qualified names.
