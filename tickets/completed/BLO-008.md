# BLO-008: GitHub templates & changelog

## Objective

Add GitHub issue templates, a pull request template, and a changelog file to standardize how issues and PRs are created and how releases are documented.

## Context

The project has no issue or PR templates, meaning contributors must guess the expected format. A changelog following the Keep a Changelog standard will document releases once the project reaches that stage. These are most useful when external contribution begins but are low-cost to set up early.

## Acceptance Criteria

- [x] `.github/ISSUE_TEMPLATE/bug_report.md` exists with YAML frontmatter (name, about, labels) and sections for: description, steps to reproduce, expected behavior, actual behavior, environment (Python version, OS, ROM if relevant), additional context
- [x] `.github/ISSUE_TEMPLATE/feature_request.md` exists with YAML frontmatter and sections for: description, motivation/use case, proposed solution, alternatives considered, relation to `docs/mcp-spec.md` tools
- [x] `.github/ISSUE_TEMPLATE/config.yml` exists with `blank_issues_enabled: true`
- [x] `.github/PULL_REQUEST_TEMPLATE.md` exists with sections for: summary, related ticket (`BLO-XXX`), type of change, testing performed, and a checklist (acceptance criteria met, tests updated, docs updated)
- [x] `CHANGELOG.md` exists in Keep a Changelog format with an `[Unreleased]` section and category headers (Added, Changed, Deprecated, Removed, Fixed, Security)

## Technical Approach

1. **Issue templates** — Use Markdown-based templates (not YAML forms) for simplicity. Bug report focuses on reproduction; feature request links to the MCP spec for context.

2. **PR template** — Reference the ticket system. Every PR must cite a `BLO-XXX` ticket ID. Include checklist items aligned with CLAUDE.md conventions.

3. **CHANGELOG** — Follow [Keep a Changelog](https://keepachangelog.com/) format. Initialize with `[Unreleased]` section only since no releases exist yet.

## Dependencies

None — these are documentation/configuration files.
