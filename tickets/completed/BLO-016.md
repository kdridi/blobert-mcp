# BLO-016: Disassembly algorithms and MCP tools

## Objective

Implement the three high-level disassembly algorithms (`disassemble_range`, `disassemble_function`, `disassemble_at_pc`) as pure domain functions, then expose them through three P0 MCP tools. After this ticket, an LLM can disassemble any range of code, trace a function from entry to return, and view code around the current program counter.

## Context

Built on top of BLO-015 (opcode table + instruction decoder). This ticket adds the algorithms and the thin MCP tool wrappers that connect them to the emulator session. The KB label enrichment (decorating output with knowledge base labels) is intentionally deferred to BLO-017, which depends on BLO-005.

Covers spec Â§1: `disassemble_range`, `disassemble_function`, `disassemble_at_pc` (all P0).

## Acceptance Criteria

- [x] `disassemble_range(memory_reader, address, length=None, end_address=None, bank=None) -> list[Instruction]` â€” decodes instructions from `address` until `length` bytes consumed, `end_address` reached, or 256-instruction cap. At least one of `length` or `end_address` is required.
- [x] `disassemble_function(memory_reader, entry_point, bank=None) -> dict` â€” traces forward from `entry_point`, stopping at unconditional `RET` (0xC9), `RETI` (0xD9), or unconditional `JP` (0xC3) that jumps outside the traced range; safety cap of 1024 instructions. Returns `{"instructions": [...], "size_bytes": N}`.
- [x] `disassemble_at_pc(memory_reader, pc, before=5, after=20) -> list[Instruction]` â€” backward-scan heuristic to find `before` instructions before `pc`: start from `pc - before * 3`, repeatedly try sequences that align with `pc`; fall back to `pc - before` if alignment fails. Marks the PC instruction with `"current": true`.
- [x] Three MCP tools (`disassemble_range`, `disassemble_function`, `disassemble_at_pc`) registered in `server.py`, follow the 5-step tool pattern, return structured dicts with instruction lists.
- [x] Integration tests cover: range decoding with explicit length and with end_address, function tracing stopping at correct terminal opcodes, at_pc backward scan alignment, tool error paths (no ROM loaded, invalid address, address out of range).

## Technical Approach

## Notes

- The `memory_reader` callable pattern keeps the disassembler fully decoupled from PyBoy and testable with synthetic bytes.
- `disassemble_at_pc` backward scan: variable-length ISA means scanning backward is ambiguous. The heuristic: try starting from `pc - before*3` bytes back, find the sequence that lands on `pc` with no remainder. If multiple starting points align, prefer the one closest to `pc`.
- Performance: disassembler is called frequently. Keep allocations minimal â€” avoid building large intermediate structures.
- No KB label enrichment in this ticket â€” the `label_resolver` parameter exists as `None` default but is never wired. BLO-017 adds the wiring after BLO-005 provides the KB.
